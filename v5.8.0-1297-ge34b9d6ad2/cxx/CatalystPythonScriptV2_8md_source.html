1 Anatomy of Catalyst Python Module (Version 2.0) {#CatalystPythonScriptsV2}
2 ===============================================
3 
4 This page describes the details of Python modules intended for use in Catalyst.
5 This page covers features supported by ParaView 5.9 and later i.e. version 2.0.
6 
7 Basics
8 -----
9 
10 Catalyst Python modules are simply Python scripts that use the `paraview`
11 package to define data analysis and visualization pipelines. They can be
12 structured as a standard `.py` file or collection of `.py` files (i.e. a Python package).
13 The former is suitable for simple analysis scripts, while the latter is intended for
14 more complex pipelines and use-cases.
15 
16 When structured as a package, the directory containing the package (and all the
17 `.py` files (modules) that comprise the package) can be zipped in to a `.zip`
18 file with same name as as the package. Directly loading such a `.zip` archive is
19 also supported by Catalyst. In fact, this is the recommended way for production
20 runs on HPC systems.
21 
22 Catalyst adaptor developers can use `vtkCPPythonScriptV2Pipeline` to add
23 vtkCPPipeline objects to the co-processor for version 2.0 Catalyst Python
24 modules structured as a single `.py` file, or a package or `.zip` containing a
25 package.
26 
27 Simulation codes that use Python for interfacing with Catalyst can leverage
28 `paraview.catalyst.bridge` to deal with all the Catalyst initialization and
29 update complexity. For such codes can use `paraview.catalyst.bridge.add_pipeline_v2`
30 to register version 2.0 Catalyst Python scripts, packages or package-zips.

33 Execution
34 ---------
35 
36 Now, let's look at how and when the Python code in the Catalyst Python module is executed.
37 The Catalyst co-processor (vtkCPProcessor) splits in situ pipeline execution into two stages:
38 first is called **RequestDataDescription**, where meta-data is collected / updated and
39 **CoProcess** where the analysis pipeline execution takes place.
40 
41 The Catalyst module gets loaded, rather *imported*, the first time
42 **RequestDataDescription** gets called. Thus, any statements that you have in
43 the module script (or `__init__.py` for packages) will get executed at that point.
44 This implies the following:
45 
46 1. Since RequestDataDescription pass is intended for gathering meta-data, it is not assured that the
47  simulation will provide any valid data at this stage. This means that any code that will get
48  executed at this stage cannot depend on simulation data to be available.
49 
50 2. If you are using Python package instead of a script or module, Python defines the entry point for the
51  package as a file named `__init__.py`. Thus, any code you have in this file will get executed.
52  Any code in other .py files in the package need not get executed unless explicitly imported in the
53  `__init__.py` file.
54 
55 At the very least, the module must define a global variable named `options` which defines co-processing
56 options like frequency of updates, output directories, etc. The following is the typical way of creating
57 and initializing this options instance.
58 
59 ```py
60 # catalyst options
61 from paraview.catalyst import Options
62 options = Options()
63 options.ExtractsOutputDirectory = "...."
64 
65 # global trigger params (optional)
66 options.GlobalTrigger.UseStartTimeStep = ... # default=False
67 options.GlobalTrigger.StartTimeStep = ... # default=0
68 options.GlobalTrigger.UseEndTimeStep = .. # default=False
69 options.GlobalTrigger.EndTimeStep = ... # default=0
70 options.GlobalTrigger.Frequency = ... # default=1
71 
72 # live params (optional)
73 options.EnableCatalystLive = ... # default=False
74 options.CatalystLiveURL = ... # default="localhost:2222"
75 options.CatalystLiveTrigger.UseStartTimeStep = ... # default=False
76 options.CatalystLiveTrigger.StartTimeStep = ... # default=0
77 options.CatalystLiveTrigger.UseEndTimeStep = ... # default=False
78 options.CatalystLiveTrigger.EndTimeStep = ... # default=0
79 options.CatalystLiveTrigger.Frequency = ... # default=1
80 
81 ```
82 
83 Once the module is imported, Catalyst evaluates the GlobalTrigger and CatalystLiveTrigger parameters
84 (if EnableCatalystLive is set to True) to determine whether to continue with subsequent steps. If
85 the trigger criteria is not statisfied further processing of the Catalyst modules is skipped until new
86 **RequestDataDescription** when the trigger criteria is reevaluated.
87 
88 Next, Catalyst checks if the module has a function called `catalyst_request_data_description`. If defined,
89 this function is called with current vtkCPDataDescription object as argument. This is only intended for
90 very custom and advanced use-cases, including testing. Generally, users should not need to rely on any
91 such `catalyst_` functions.
92 
93 After **RequestDataDescription** stage, the Catalyst co-processor will trigger **CoProcess** stage, unless
94 it was skipped (based on GlobalTrigger and CatalystLiveTrigger parameters). In **CoProcess**, simulation has
95 made its data available for analysis. Thus code can now rely on that for setup or execution.
96 First time **CoProcess** gets called, Catalyst looks for presence of a global variable named **scripts**
97 in the module. When present, this is a list of names of submodules that have code to setup analysis
98 and visualization pipelines. Typically, these are simply ParaView Python scripts that define the visualization
99 state -- same as the scripts used for `pvpython` or `pvbatch`. All submodules list in **scripts** variable
100 are imported at this point. Since this happens during **CoProcess**, these scripts can safely rely on
101 valid simulation data being present to setup pipeline state e.g. setting up filter parameters,
102 color map ranges, etc. Note, this happens only the first time **CoProcess** is called, i.e. subsequent
103 **CoProcess** calls don't cause the analysis scripts to be reimported.
104 
105 ```py
106 # this will cause a `pipeline.py` from the Python package to be imported
107 # in first call to CoProcess
108 scripts = ["pipeline"]
109 ```
110 
111 The submodule may define visualization state that includes extract generators for generating extracts.
112 These extract generators may have their own triggers. Before proceeding further, Catalyst now checks if
113 any triggers defined are activated for the current timestep. If not, the further processing of the module
114 is skipped.
115 
116 Next, when not skipped, Catalyst checks for a function called `catalyst_initialize` in the module and all of
117 the submodules imported via `scripts`. If present, this function is called, again with the current
118 vtkCPDataDescription object as argument starting with the module and then on each of the submodules in the
119 same order as in the `scripts` list. This is only called once; subsequent calls to **CoProcess** will skip this
120 step. When not using a package, for example, `catalyst_initialize` is a good function to implement code to setup
121 you analysis pipeline since its called once and you're assured that the simulation data is valid -- which is often
122 necessary when setting up visualization pipelines.
123 
124 Next, Catalyst optionally calls the function `catalyst_coprocess`, if present, on the module and all
125 the submodules imported via `scripts` in same order as `catalyst_initialize`. Again, this is generally
126 not needed except for highly customized / advanced use-cases.
127 
128 Next, Catalyst will generate extracts using extract generators created by the module (or its submodules). Finally,
129 if `EnableCatalystLive` is true and the CatalystLiveTrigger is satisfied, Catalyst attempts to connect to
130 ParaView GUI at the `CatalystLiveURL` for Live.
131 
132 These **RequestDataDescription** and **CoProcess** stages are repeated for the entire simulation run. On termination,
133 Catalyst optionally calls `catalyst_finalize` on the module and submodules imported via `scripts`. Here, however, the
134 submodules are finalized before that the top-level module/package is finalized.

137 Visualization Pipeline
138 -----------------------
139 
140 ParaView Python API, exposed via the `paraview.simple` module, is used to define the visualization and data
141 pipeline. The API mimics actions one would take the GUI. For example, simple script to create a **Wavelet** source
142 slice it, looks as follows:
143 
144 ```py
145 from paraview.simple import *
146 
147 wavelet1 = Wavelet(registrationName='Wavelet1')
148 
149 # create a new 'Slice'
150 slice1 = Slice(registrationName='Slice1', Input=wavelet1)
151 slice1.SliceType = 'Plane'
152 slice1.SliceType.Normal = [0, 0, 1]
153 
154 ```
155 
156 This same code can be used in a Catalyst Python script to setup the visualization
157 pipeline. The only thing to note is that we need a mechanism to indicate which of
158 the data-producers in the pipeline should be replaced by the data generated by the
159 simulation. In Catalyst, the data produced by the simulation is available on named
160 channels. Thus, we need to identify which named-channel corresponds to which data
161 data-producer in the pipeline. For that, we use the `registrationName` attribute.
162 In the Python script, when creating a source or a filter, one can pass in an
163 optional argument `registrationName`. When the script is being executed within
164 Catalyst, Catalyst looks to see if the `registrationName` matches the name of a
165 known channel. If so, the data producer type requested (in this case Wavelet) is
166 ignored and instead is replaced by a producer that puts out simulation data on
167 that channel.
168 
169 The `paraview.demos.wavelet_miniapp` is a miniapp that acts as a simulation
170 producing a time value `vtkImageData` dataset. So we can use it to run the above
171 script.
172 
173 To do that, let's create a Python package. Create a directory, say
174 `/tmp/sample` with the following contents:
175 
176 ```
177 /tmp/sample/
178 /tmp/sample/__init__.py
179 /tmp/sample/pipeline.py
180 ```
181 
182 The `__init__.py` looks as follows:
183 ```py
184 from paraview.catalyst import Options
185 options = Options()
186 
187 scripts = ['pipeline']
188 ```
189 
190 The `pipeline.py` is simply the script shown earlier to create
191 Wavelet and slice it.
192 
193 Now, we run the miniapp with this analysis module as follows:
194 
195 ```bash
196 > pvpython -m paraview.demos.wavelet_miniapp \
197  --script /tmp/sample
198 ```
199 
200 This will produce the following output
201 ```
202 ( 0.467s) [main thread ] bridge.py:18 WARN| Warning: ParaView has been initialized before `initialize` is called
203 timestep: 1/100
204 ( 1.942s) [main thread ] detail.py:86 WARN| script may not depend on simulation data; is that expected?
205 timestep: 2/100
206 ( 1.994s) [main thread ] detail.py:86 WARN| script may not depend on simulation data; is that expected?
207 ....
208 ```
209 
210 The warning message *"script may not depend on simulation data; is that expected?"* tells us that
211 the script it not affected by any data the simulation is producing since none of
212 its data producers have been replaced by a named channel. The default channel
213 name that `wavelet_miniapp` uses is **input**. We can either change the
214 `registrationName` for the `Wavelet` source in `pipeline.py` to
215 `input` or pass optional argument `-c` to the `wavelet_miniapp` which lets us
216 rename the channel as follows:
217 
218 ```bash
219 > ./bin/pvpython -m paraview.demos.wavelet_miniapp --script-version 2 -s /tmp/sample -c Wavelet1
220 ( 0.458s) [main thread ] bridge.py:18 WARN| Warning: ParaView has been initialized before `initialize` is called
221 timestep: 1/100
222 timestep: 2/100
223 timestep: 3/100
224 ...
225 
226 ```
227 
228 The script here does nothing significant so we don't see any results. Here's a
229 tweak to the `pipeline.py` to make it show the rendering results for the slice.
230 
231 ```py
232 # pipeline.py
233 
234 from paraview.simple import *
235 
236 wavelet1 = Wavelet(registrationName='Wavelet1')
237 
238 slice1 = Slice(registrationName='Slice1', Input=wavelet1)
239 slice1.SliceType = 'Plane'
240 slice1.SliceType.Normal = [0, 0, 1]
241 
242 Show()
243 view = Render()
244 
245 def catalyst_coprocess(*args):
246  global view
247  ResetCamera(view)
248  Render(view)
249 ```
250 
251 Now, when you launch the `wavelet_miniapp`, it will show the rendering results
252 on the screen as the simulation advances.
253 
254 In this example, we are using `catalyst_coprocess` callback described earlier to
255 execute certain actions per timestep. A more advanced analysis script would use
256 extract generators, in which case `catalyst_coprocess` is rarely needed.
257 
258 Also note, in this example we simply use a directory for the Python package
259 instead of archiving it in a `.zip`. This is useful for development and
260 debugging purposes. However, for HPC / production runs, it's recommended that
261 you create an archive once the package is finalized.

263 Debugging
264 ---------
265 
266 The `pvpython`/`pvbatch` executables can be launched with a `-l` argument to
267 generate detailed logs as follows:
268 
269 ```bash
270 > pvpython -l=/tmp/log.txt,TRACE -m paraview.demos.wavelet_miniapp ...
271 ```
272 
273 Here, a `/tmp/log.txt` will all logging output will be generated. You can
274 also elevate Catalyst-generated log to a higher level and log that, for example,
275 the following will elevate catalyst log level to INFO and log all INFO messages
276 to `/tmp/infolog.txt`
277 
278 ```bash
279 > env PARAVIEW_LOG_CATALYST_VERBOSITY=INFO \
280  pvpython -l=/tmp/infolog.txt,INFO -m paraview.demos.wavelet_miniapp ...
281 ```
