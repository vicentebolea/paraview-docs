1 # Plugin Howto {#PluginHowto}
2 
3 ## Introduction
4 
5 ParaView comes with plethora of functionality bundled in: several readers,
6 multitude of filters, different types of views, etc. However, it is not
7 uncommon for developers to want to add new functionality to ParaView to, for
8 example, add support to their new file format or incorporate a new filter into
9 ParaView. ParaView makes it possible to add new functionality by using an
10 extensive plugin mechanism.
11 
12 Plugins can be used to extend ParaView in several ways:
13 
14  * Add new readers, writers, filters
15  * Add custom GUI components such as toolbar buttons to perform common tasks
16  * Add new views in for display data
17 
18 Examples for different types of plugins are provided with the ParaView source
19 under `Examples/Plugins/`.
20 
21 This document has major sections:
22 
23  * First section covers how to use existing plugins in ParaView.
24  * Second section contains information for developers about writing new
25  plugins for ParaView.
26 
27 ## Using Plugins
28 
29 ### Types of plugins
30 
31 Plugins are distributed as shared libraries (`*.so` on Unix and macOS, and
32 `*.dll` on Windows). For a plugin to be loadable in ParaView, it must be built
33 with the same version of ParaView as it is expected to be deployed on. Plugins
34 can be classified into two broad categories:
35 
36  * *Server-side plugins*: These are plugins that extend the algorithmic
37  capabilities for ParaView. For example, new filters, readers or writers.
38  Since ParaView processes data on the server-side, these plugins need to be
39  loaded on the server.
40  * *Client-side plugins*: These are plugins that extend the ParaView GUI
41  including property panels for new filters, toolbars or views. These plugins
42  need to be loaded on the client.
43 
44 Oftentimes a plugin has both server-side as well as client-side components to
45 it. For example, a plugin that adds a new filter and a property panel that goes
46 with that filter. Such plugins need to be loaded both on the server as well as
47 the client.
48 
49 Generally, users don't have to worry whether a plugin is a server-side or
50 client-side plugin. Simply load the plugin on the server as well as the client.
51 ParaView will include relevant components from plugin on each of the processes.
52 
53 ### Loading plugins
54 
55 There are four ways for loading plugins:
56 
57  * *Using the GUI* (`Plugin Manager`)
58  - Plugins can be loaded into ParaView using the `Plugin Manager` accessible
59  from the `Tools | Manage Plugins/Extensions` menu. The `Plugin Manager`
60  has two sections for loading client plugins and server plugins (shown
61  only when connected to a server). To load a plugin on the client and
62  server side, simply browse to the plugin shared library. If the plugin is
63  loaded successfully, it will appear in the list of loaded plugins. The
64  `Plugin Manager` also lists the paths it searched to load plugins
65  automatically.
66  - The `Plugin Manager` remembers all loaded plugins across ParaView
67  instances, so once a plugin is loaded once, it will appear in the future
68  (unloaded).
69  - You can set up ParaView to automatically load the plugin at startup (for
70  client plugins) or on connecting to the server (for server plugins) by
71  checking the "Auto Load" checkbox on a loaded plugin.
72 
73 ![Plugin Manager when not connected to a remote server, showing loaded plugins on the local site.](images/LocalPlugin_Manager.png)
74 ![Plugin Manager when connected to a server showing loaded plugins on the local as well as remote sites.](images/RemotePlugin_Manager.png)
75 
76  * *Using environment variable* (Auto-loading plugins)
77  - In order to have ParaView automatically load a set of plugins on startup,
78  one can use the `PV_PLUGIN_PATH` environment variable. `PV_PLUGIN_PATH`
79  may be used to list a set of directories (separated by colon (`:`) for
80  Unix platforms or semi-colon (`;`) on Windows) which ParaView will search
81  on startup to load plugins. This environment variable needs to be set on
82  both the client and server sides to load their respective plugins. Note
83  that plugins in PV_PLUGIN_PATH are always auto-loaded irrespective of the
84  status of the `Auto Load` checkbox in the `Plugin Manager`. Paths in this
85  list may also be of the structure created by the ParaView plugin macros
86  (e.g., `MyPlugin/MyPlugin.so`).
87  - Finer control can be used using the `PV_PLUGIN_CONFIG_FILE` environment
88  variable. `PV_PLUGIN_CONFIG_FILE` can be used to list a set of XML plugin
89  configuration files (separated by colon (`:`) on Unix platforms or
90  semi-colon (`;`) on Windows). ParaView will read these files on startup
91  to load specified plugins. The XML plugin configuration file format looks
92  like this:
93 
94 ```xml
95 <?xml version="1.0"?>
96 <Plugins>
97  <Plugin name="MyPlugin" filename="/absolute/path/to/libMyPlugin.so"/>
98  <!-- Note that relative paths are calculated from the directory of this XML file. -->
99  <Plugin name="MyPluginRel" filename="relative/path/to/libMyPlugin.so"/>
100 </Plugins>
101 ```
102 
103  Plugins listed this way will always be loaded, irrespective of the status
104  of the `Auto Load` checkbox in the `Plugin Manager`.
105  * *Using the plugin file `.plugins`* (Make plugins available and possibly
106  Auto-load plugins)
107  - Plugins that are listed in the `.plugins` file on the client and server
108  will automatically be listed in the `Plugin Manager`, and may optionally
109  be auto loaded. ParaView creates its own `.plugins` file listing plugins
110  known during its build and uses it as the default. An example `.plugins`
111  file, auto loading H5PartReader, looks like this:
112 
113 ```xml
114 <?xml version="1.0"?>
115 <Plugins>
116  <Plugin name="Moments" auto_load="0"/>
117  <Plugin name="PrismPlugin" auto_load="0"/>
118  <Plugin name="PointSprite_Plugin" auto_load="0"/>
119  <Plugin name="pvblot" auto_load="0"/>
120  <Plugin name="H5PartReader" auto_load="1"/>
121 </Plugins>
122 ```
123 
124  * *Default search paths*
125  - Recognized locations are:
126  * A `plugins` subdirectory under the `paraview-X.Y` directory in the
127  library path (usually `lib` on Unix platforms and `bin` on Windows).
128  * A `plugins` subdirectory in the user's home directory. On Unix
129  platforms, `$HOME/.config/ParaView/ParaView<version>/Plugins`. On
130  Windows `%APPDATA$\ParaView\ParaView<version>\Plugins`. (XXX: no
131  evidence of an implementation backing this search path).
132

134

156 # Writing plugins
157 
158 This section covers writing and compiling different types of Plugins. To create
159 a plugin, one must have their own build of ParaView. Binaries downloaded from
160 www.paraview.org do not include necessary header files or import libraries
161 (where applicable) for compiling plugins.
162 
163 The `CMakeLists.txt` file used in all following examples start off with the
164 following code:
165 
166 ```cmake
167 # ParaView requires CMake 3.8 in order to be used.
168 cmake_minimum_required(VERSION 3.8)
169 project(myplugin C CXX)
170 
171 find_package(ParaView REQUIRED)
172 ```
173 
174 Where CMake will ask for the `ParaView_DIR` which you point to the ParaView
175 build or install tree you would to build your with.
176 
177 Note that the `C` and `CXX` languages are required in general because ParaView
178 may need to find other packages which are written with only C in mind (MPI is
179 the usual culprit here) and need to know about the C compiler that is
180 available.
181 
182 ## Exposing an Existing Filter
183 
184 Sometimes, the filter that one wants to add to ParaView is already available in
185 VTK, it's just not exposed through the ParaView GUI. This is the easiest type
186 of plugin to create. There are two options:
187 
188  1. setup the plugin using only an XML file; and
189  2. actually compile the plugin into a shared library.
190 
191 The first option is the easiest, but the second option will prepare you for
192 creating a custom filter in the future as the process is nearly identical.
193 
194 ## Adding a New Filter
195 
196 It is also possible to add new filters to ParaView. The filter has to be a
197 VTK-based algorithm, written as following the standard procedures for writing
198 VTK algorithms. Generally for such cases where we are adding a new VTK class to
199 ParaView (be it a filter, reader or a writer), we need to do the following
200 tasks:
201 
202  * Write a *Server Manager Configuration XML* which describes the `Proxy`
203  interface for the filter. Basically, this defines the interface for the
204  client to create and modify instances of the new class on the server side.
205  Please refer to the [ParaView Guide][] for details about writing these
206  server-manager XMLs.
207  * Write a configuration XML for the GUI to make ParaView GUI aware of this
208  new class, if applicable. For filters, this is optional, since ParaView
209  automatically recognizes filters added through plugins and lists them in
210  the *Alphabetical* sub-menu. One may use the GUI configuration XML to add
211  the new filter to a specific category in the *Filters* menu, or add a new
212  category. For readers and writers, this is required since ParaView GUI
213  needs to know what extensions your reader/writer supports etc.
214 
215 ## Examples
216 
217 ### XML Plugins
218 
219 If you have not built ParaView from source, using an XML plugin is your only
220 option.
221 
222 First, a server manager XML for the filter is required. The GUI XML to add the
223 filter to any specific category is optional.
224 
225 For example, let's say we simply want to expose the `vtkCellDerivatives` filter
226 in VTK. Then first, we'll write the server manager configuration XML (call it
227 `CellDerivatives.xml`), similar to what we would have done for adding a new
228 filter.
229 
230 ```xml
231 <ServerManagerConfiguration>
232  <ProxyGroup name="filters">
233  <SourceProxy name="MyCellDerivatives" class="vtkCellDerivatives" label="My Cell Derivatives">
234  <Documentation
235  long_help="Create point attribute array by projecting points onto an elevation vector."
236  short_help="Create a point array representing elevation.">
237  </Documentation>
238  <InputProperty
239  name="Input"
240  command="SetInputConnection">
241  <ProxyGroupDomain name="groups">
242  <Group name="sources"/>
243  <Group name="filters"/>
244  </ProxyGroupDomain>
245  <DataTypeDomain name="input_type">
246  <DataType value="vtkDataSet"/>
247  </DataTypeDomain>
248  </InputProperty>
249  </SourceProxy>
250  </ProxyGroup>
251 </ServerManagerConfiguration>
252 ```
253 
254 At this point, we can stop and use the plugin in ParaView by loading the XML
255 file directly into the plugin manager.
256 
257 Please note that if you are writing the XML for a filter that takes just one
258 input, you *must* set the `name` attribute for the `InputProperty` XML element
259 to `Input`. If you do not, then the filter will not be displayed properly in
260 ParaView's pipeline browser.
261 
262 ### Compiling into a Shared Library
263 
264 If you have built ParaView from source, it is possible to compile the plugin
265 into into a shared library. To do this, we can use the following top-level:
266 `CMakeLists.txt`:
267 
268 ```cmake
269 # Standard CMake boilerplate. ParaView's `find_package` requires at least 3.8.
270 cmake_minimum_required(VERSION 3.8)
271 project(sharedlibrary)
272 
273 # These five lines are required in order to set up installation directories
274 # (which also control build directory locations) and enable shared builds
275 # (CMake's default is for a static build).
276 include(GNUInstallDirs)
277 set(CMAKE_RUNTIME_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/${CMAKE_INSTALL_BINDIR}")
278 set(CMAKE_LIBRARY_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/${CMAKE_INSTALL_LIBDIR}")
279 set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/${CMAKE_INSTALL_LIBDIR}")
280 set(BUILD_SHARED_LIBS ON)
281 
282 # Find ParaView. This will bring in ParaView's CMake API and imported targets.
283 find_package(ParaView REQUIRED)
284 
285 # Scan the plugin file in order to set up internal data structures for building
286 # plugins.
287 paraview_plugin_scan(
288  # The `paraview.plugin` file describing the plugin.
289  PLUGIN_FILES "${CMAKE_CURRENT_SOURCE_DIR}/Plugin/paraview.plugin"
290  # A result variable for the (enabled) plugins found during the scan.
291  PROVIDES_PLUGINS plugins
292  # Enable plugins during this scan by default.
293  ENABLE_BY_DEFAULT ON)
294 
295 # Build the plugins discovered during the scan.
296 paraview_plugin_build(
297  PLUGINS ${plugins})
298 ```
299 
300 The mentioned `paraview.plugin` file describes the plugin to the build system:
301 
302 ```cmake
303 NAME
304  CellDerivatives
305 DESCRIPTION
306  Expose the vtkCellDerivatives class to ParaView.
307 REQUIRES_MODULES
308  # This module provides the `vtkCellDerivatives` filter.
309  VTK::FiltersGeneral
310 ```
311 
312 In the `Plugin` directory (beside the `paraview.plugin` file), the plugin is
313 given the information it needs to build:
314 
315 ```cmake
316 paraview_add_plugin(CellDerivatives
317  VERSION "1.0"
318  SERVER_MANAGER_XML CellDerivatives.xml)
319 ```
320 
321 Then using CMake, one can build a plugin for this new filter. We can now load
322 the plugin through the plugin manager by selecting the created `.so` or `.dll`
323 file.
324 
325 ### Qt resource plugins
326 
327 Similarly compiled Qt resources (`*.bqrc`) can be loaded at runtime. A `.bqrc`
328 file is a binary file containing resources which can include icons, the GUI
329 configuration XML for adding categories, etc. A `.bqrc` can be made from a
330 `.qrc` by running the `rcc` utility provided by Qt:
331 
332 ```sh
333 rcc -binary -o myfile.bqrc myfile.qrc
334 ```

336 ### Adding a New Filter
337 
338 For this example, refer to `Examples/Plugins/ElevationFilter` in the ParaView
339 source. Let's say we have written a new `vtkMyElevationFilter`
340 (`vtkMyElevationFilter.{h,cxx}`), which extends the functionality of the
341 `vtkElevationFilter` and we want to package that as a plugin for ParaView. For
342 starters, we simply want to use this filter in ParaView (e.g., not doing
343 anything fancy with *Filters* menu categories). As described, we need to write
344 the server manager configuration XML (`MyElevationFilter.xml`). Once that's
345 done, we write a `CMakeLists.txt` file to package this into a plugin.
346 
347 This `CMakeLists.txt` needs to include the following lines:
348 
349 ```cmake
350 cmake_minimum_required(VERSION 3.8)
351 project(newfilter)
352 
353 include(GNUInstallDirs)
354 set(BUILD_SHARED_LIBS ON)
355 
356 find_package(ParaView REQUIRED)
357 
358 paraview_plugin_scan(
359  PLUGIN_FILES "${CMAKE_CURRENT_SOURCE_DIR}/Plugin/paraview.plugin"
360  PROVIDES_PLUGINS plugins
361  ENABLE_BY_DEFAULT ON)
362 
363 paraview_plugin_build(
364  PLUGINS ${plugins})
365 ```
366 
367 The referenced `paraview.plugin` file contains:
368 
369 ```cmake
370 NAME
371  ElevationFilter
372 DESCRIPTION
373  An example paraview plugin containing server manager XML and the server
374  manager classes to build. This plugin can be loaded on the server side.
375 REQUIRES_MODULES
376  VTK::CommonCore
377  VTK::FiltersCore
378 ```
379 
380 And the `CMakeLists.txt` file beside it contains:
381 
382 ```cmake
383 paraview_add_plugin(ElevationFilter
384  VERSION "1.0"
385  MODULES ElevationFilters
386  MODULE_FILES "${CMAKE_CURRENT_SOURCE_DIR}/ElevationFilters/vtk.module")
387 ```
388 
389 Because we are building our own custom filter, it needs to be a VTK module in
390 order to support having its information available to the XML code. First, the
391 module is declared in a `vtk.module` file:
392 
393 ```cmake
394 NAME
395  ElevationFilters
396 DEPENDS
397  VTK::FiltersCore
398 PRIVATE_DEPENDS
399  VTK::CommonCore
400 ```
401 
402 And then the module is built with its associated server manager XML file
403 attached to the module. Note that the module name cannot be the same as the
404 plugin name due to the way the library targets are managed internally.
405 
406 ```cmake
407 set(classes
408  vtkMyElevationFilter)
409 
410 # Find external packages here using `find_package`.
411 
412 vtk_module_add_module(ElevationFilters
413  CLASSES ${classes})
414 
415 # Link to external packages here using `vtk_module_link(ElevationFilters)`.
416 
417 paraview_add_server_manager_xmls(
418  XMLS MyElevationFilter.xml)
419 ```
420 
421 Then using CMake, one can build a plugin for this new filter. Once this plugin
422 is loaded the filter will appear under the *Alphabetical* list in the *Filters*
423 menu. Note that there will be two libraries in the resulting directory. Be sure
424 to load the `ElevationFilter` one which is the plugin, not the
425 `ElevationFilters` module library.
426 
427 ### Filters with Multiple Input Ports
428 
429 If a filter requires multiple input ports, there are two options:
430 
431  1. Create helper functions in the VTK filter such as `SetYourInputName` which
432  deal with addressing the VTK pipeline in the C++ code; and
433  2. Address/access the input connection by number in the XML. The `port_index`
434  property specifies which input connection the particular input will be
435  connected to. The `SetInputConnection` function is the command that will
436  actually be called with this `port_index` to setup the pipeline.
437 
438 An example XML file for a filter with multiple inputs is below. The filter
439 takes three `vtkPolyData` objects as input.
440 
441 ```xml
442 <ServerManagerConfiguration>
443  <ProxyGroup name="filters">
444  <SourceProxy name="LandmarkTransformFilter" class="vtkLandmarkTransformFilter" label="LandmarkTransformFilter">
445  <Documentation
446  long_help="Align two point sets using vtkLandmarkTransform to compute the best transformation between the two point sets."
447  short_help="vtkLandmarkTransformFilter.">
448  </Documentation>
449 
450  <InputProperty
451  name="SourceLandmarks"
452  port_index="0"
453  command="SetInputConnection">
454  <ProxyGroupDomain name="groups">
455  <Group name="sources"/>
456  <Group name="filters"/>
457  </ProxyGroupDomain>
458  <DataTypeDomain name="input_type">
459  <DataType value="vtkPolyData"/>
460  </DataTypeDomain>
461  <Documentation>
462  Set the source data set. This data set that will move towards the target data set.
463  </Documentation>
464  </InputProperty>
465 
466  <InputProperty
467  name="TargetLandmarks"
468  port_index="1"
469  command="SetInputConnection">
470  <ProxyGroupDomain name="groups">
471  <Group name="sources"/>
472  <Group name="filters"/>
473  </ProxyGroupDomain>
474  <DataTypeDomain name="input_type">
475  <DataType value="vtkPolyData"/>
476  </DataTypeDomain>
477  <Documentation>
478  Set the target data set. This data set will stay stationary.
479  </Documentation>
480  </InputProperty>
481 
482  <InputProperty
483  name="SourceDataSet"
484  port_index="2"
485  command="SetInputConnection">
486  <ProxyGroupDomain name="groups">
487  <Group name="sources"/>
488  <Group name="filters"/>
489  </ProxyGroupDomain>
490  <DataTypeDomain name="input_type">
491  <DataType value="vtkPolyData"/>
492  </DataTypeDomain>
493  <Documentation>
494  Set the source data set landmark points.
495  </Documentation>
496  </InputProperty>
497 
498  <Hints>
499  <!-- see below for what options to put here -->
500  </Hints>
501 
502  </SourceProxy>
503  </ProxyGroup>
504 </ServerManagerConfiguration>
505 ```
506 
507 To set the inputs in ParaView, simply select one of the inputs in the *Pipeline
508 Browser* and then select the filter from the *Filters* menu. This will open a
509 dialog box which will allow you to specify which object to connect to each
510 input port.
511 
512 ### Adding *Categories* to the *Filters* Menu
513 
514 Now suppose we want to add a new category to the *Filters* menu, called
515 *Extensions* and then show this filter in that menu. In that case we need to
516 add a hint to the XML file that tells ParaView what category to display this
517 filter in. In this case, the `Hints` element of the XML file can contain:
518 
519 ```xml
520 <Hints>
521  <ShowInMenu category="Extensions" />
522 </Hints>
523 ```
524 
525 If the name of the category is same as an already existing category such as
526 *Data Analysis*, then the filter gets added to the existing category.
527 
528 ### Adding Icons
529 
530 You can see that some filters in the *Filters* menu (e.g., *Clip*) have icons
531 associated with them. It's possible for the plugin to add icons for filters it
532 adds as well. For that you need to write a Qt resource file (say
533 `MyElevation.qrc`) as follows:
534 
535 ```xml
536 <RCC>
537  <qresource prefix="/MyIcons" >
538  <file>MyElevationIcon.png</file>
539  </qresource>
540 </RCC>
541 ```
542 
543 To use the icon for a filter in the pipeline add the following hint to the
544 server manager XML.
545 
546 ```xml
547 <Hints>
548  <ShowInMenu icon=":/MyIcons/MyElevationIcon.png" />
549 </Hints>
550 ```
551 
552 Finally, the plugin's `CMakeLists.txt` file much change to include our
553 `MyElevation.qrc` file as follows:
554 
555 ```cmake
556 paraview_add_plugin(ElevationFilter
557  VERSION "1.0"
558  MODULES ElevationFilters
559  MODULE_FILES "${CMAKE_CURRENT_SOURCE_DIR}/ElevationFilters/vtk.module"
560  UI_RESOURCES MyElevation.qrc)
561 ```

563 ### Adding GUI Parameters
564 
565 Simply add these in the server manager XML to expose parameters of the filter
566 to the ParaView user.
567 
568 #### Integer Property
569 
570 This property appears as a text box.
571 
572 ```xml
573 <IntVectorProperty
574  name="bStartByMatchingCentroids"
575  command="SetbStartByMatchingCentroids"
576  number_of_elements="1"
577  default_values="1">
578 </IntVectorProperty>
579 ```
580 
581 #### Boolean Property
582 
583 This property appears as a check box control. A boolean property uses the
584 `IntVectorProperty` with an extra line (`BooleanDomain`) indicating this should
585 be a check box rather than a text field.
586 
587 ```xml
588 <IntVectorProperty
589  name="bStartByMatchingCentroids"
590  command="SetbStartByMatchingCentroids"
591  number_of_elements="1"
592  default_values="1">
593  <BooleanDomain name="bool"/>
594 </IntVectorProperty>
595 ```
596 
597 #### String Property
598 
599 This property appears as a text box.
600 
601 ```xml
602 <StringVectorProperty
603  name="YourStringVariable"
604  command="SetYourStringVariable"
605  number_of_elements="1"
606  default_values="1">
607 </StringVectorProperty>
608 ```
609 
610 #### Double Property
611 
612 This property appears as a text box.
613 
614 ```xml
615 <DoubleVectorProperty
616  name="YourDoubleVariable"
617  command="SetYourDoubleVariable"
618  number_of_elements="1"
619  default_values="1">
620 </DoubleVectorProperty>
621 ```
622 
623 #### Multi-Value Double Property
624 
625 This property appears as a text box.
626 
627 ```xml
628 <DoubleVectorProperty
629  name="YourDoubleVectorVariable"
630  command="SetYourDoubleVectorVariable"
631  number_of_elements="3"
632  default_values="1.0 0.0 0.0">
633 </DoubleVectorProperty>
634 ```
635 
636 #### Double Property Slider
637 
638 This creates a slider that ranges from `0.0` to `1.0`.
639 
640 ```xml
641 <DoubleVectorProperty
642  name="PercentToRemove"
643  command="SetPercentToRemove"
644  number_of_elements="1"
645  default_values="0.1">
646  <DoubleRangeDomain name="range" min="0.0" max="1.0" />
647 </DoubleVectorProperty>
648 ```
649 
650 #### Drop Down List
651 
652 This creates a drop down list with 3 choices. The values associated with the
653 choices may be specified.
654 
655 ```xml
656 <IntVectorProperty
657  name="TransformMode"
658  command="SetTransformMode"
659  number_of_elements="1"
660  default_values="1">
661  <EnumerationDomain name="enum">
662  <Entry value="6" text="RigidBody"/>
663  <Entry value="7" text="Similarity"/>
664  <Entry value="12" text="Affine"/>
665  </EnumerationDomain>
666  <Documentation>
667  This property indicates which transform mode will be used.
668  </Documentation>
669 </IntVectorProperty>
670 ```
671 
672 #### Drop Down List with Values from Input Arrays
673 
674 This creates a list that lets you choose among the input arrays of the input of
675 a `ProgrammableFilter`:
676 
677 ```xml
678 <StringVectorProperty
679  name="SelectInputScalars"

766 #### Adding a Reader
767 
768 Adding a new reader through a plugin is similar to adding a filter. The only
769 difference is that we do not need to specify what category the reader should be
770 added to in the GUI. For the latest version of ParaView we do not need to
771 specify anything special for the GUI as all of the details of the reader are
772 available in the XML proxy definition of the reader. For ParaView version 4.0.1
773 and earlier we need the XML to define what file extensions this reader can
774 handle. This XML (`MyReaderGUI.xml`) looks like this:
775 
776 ```xml
777 <ParaViewReaders>
778  <Reader name="MyPNGReader" extensions="png"
779  file_description="My PNG Files">
780  </Reader>
781 </ParaViewReaders>
782 ```
783 
784 An example `MyPNGReader.xml` is shown below. In almost all cases you must have
785 a `SetFileName` function property. You are free to have other properties as
786 well, as with a standard (non-reader) filter. Also, the `Hints` section is needed
787 in order to associate the file extension with the reader on the client. The
788 `ReaderFactory` hint is what the client uses to identify readers from sources.
789 
790 ```xml
791 <ServerManagerConfiguration>
792  <ProxyGroup name="sources">
793  <SourceProxy name="MyPNGReader" class="vtkMyPNGReader" label="PNGReader">
794  <Documentation
795  long_help="Read a PNG file."
796  short_help="Read a PNG file.">
797  </Documentation>
798  <StringVectorProperty
799  name="FileName"
800  animateable="0"
801  command="SetFileName"
802  number_of_elements="1">
803  <FileListDomain name="files"/>
804  <Documentation>
805  This property specifies the file name for the PNG reader.
806  </Documentation>
807  </StringVectorProperty>
808  <Hints>
809  <ReaderFactory extensions="png"
810  file_description="PNG File Format" />
811  </Hints>
812  </SourceProxy>
813  </ProxyGroup>
814 </ServerManagerConfiguration>
815 ```
816 
817 The CMake code for a reader plugin uses the same structure as the filter
818 example. The only likely difference is that the plugin should also pass
819 `REQUIRED_ON_SERVER` to `paraview_add_plugin` since the server side needs the
820 reader available for its use.
821 
822 If you want your reader to work correctly with a file series, please refer to [[Animating legacy VTK file series#Making custom readers work with file series|file series animation]] for details.
823 
824 Once you generate the project using CMake and compile the project, in ParaView
825 go to *Tools > Manage Plugins/Extensions*. Under *Local Plugins*, click *Load
826 New* and browse for the shared library file you just created. You should now
827 see your new file type in the *Files of type* list in the *Open file* dialog.

842 #### Adding Customizations for Properties Panel
843 
844 <!-- TODO link to external page -->
845 
846 [[ParaView/Properties Panel|Properties Panel]] is the primary panel in ParaView
847 used to change the parameters for visualization modules and displays. Plugins
848 can provide new types of [`pqPropertyWidget`][pqPropertyWidget] subclasses that
849 can be used to control properties/property groups on this Properties panel.
850 
851 To register a new `pqPropertyWidget` subclass to be associated with a
852 particular widget type for a property (`vtkSMProperty`), use the following
853 CMake code in your plugin:
854 
855 ```cmake
856 paraview_plugin_add_property_widget
857  KIND kind
858  TYPE my_property_widget_type
859  CLASS_NAME ClassName
860  INTERFACES interfaces
861  SOURCES sources)
862 
863 paraview_add_plugin(propwidget
864  VERSION "1.0"
865  UI_INTERFACES ${interfaces}
866  SOURCES ${sources})
867 ```
868 
869 The `KIND` argument must be one of `WIDGET`, `GROUP_WIDGET`, or
870 `WIDGET_DECORATOR`. For a `vtkSMProperty`, `WIDGET` is required.
871 
872 The `CLASS_NAME` argument must refer to a `pqPropertyWidget` subclass with a
873 constructor with the following signature:
874 
875 ```cpp
876 ClassName(vtkSMProxy *smproxy, vtkSMProperty *smproperty, QWidget *parentObject)
877 ```
878 
879 The `TYPE` argument specifies the string that will be used in the server
880 manager XML as the value for the `panel_widget` attribute to request creation
881 of this widget for a `vtkSMProperty` subclass.
882 
883 To register a new `pqPropertyWidget` subclass to be associated with a
884 particular widget type for a property group (`vtkSMPropertyGroup`), use
885 `GROUP_WIDGET` for the `KIND` argument. The referenced `CLASS_NAME` must
886 subclass `pqPropertyWidget` and have a constructor with the signature:
887 
888 ```cpp
889 ClassName(vtkSMProxy *smproxy, vtkSMPropertyGroup *smgroup, QWidget *parentObject);
890 ```
891 
892 As before, the `TYPE` specifies the string that will be used in the server
893 manager XML as the value for the `panel_widget` attribute on a `PropertyGroup`
894 element to request creation of this widget for that group.
895 
896 Another mechanism for adding customizations for *Properties* panel is to
897 provide [`pqPropertyWidgetDecorator`][pqPropertyWidgetDecorator] subclasses to
898 add custom control logic for widgets on the panel.
899 
900 Decorators use the `WIDGET_DECORATOR` argument to `KIND`.
901 
902 The `CLASS_NAME` must point to a `pqPropertyWidgetDecorator` subclass and the
903 `TYPE` is the string name used to request the creation of the decorator in the
904 server manager XML as described [[ParaView/Properties Panel|here]].
905 
906 An example for customizing the Properties panel can be found in the ParaView
907 source under `Examples/Plugins/PropertyWidgets`.

937 #### Adding a Toolbar
938 
939 Filters, reader, and writers are by far the most common ways for extending
940 ParaView. However, ParaView plugin functionality goes far beyond that. The
941 following sections cover some of these advanced plugins that can be written.
942 
943 Applications use toolbars to provide easy access to commonly used
944 functionality. It is possible to have plugins that add new toolbars to
945 ParaView. The plugin developer implements his own C++ code to handle the
946 callback for each button on the toolbar. Hence one can do virtually any
947 operation using the toolbar plugin with some understanding of the ParaView
948 Server Manager framework and the ParaView GUI components.
949 
950 Please refer to `Examples/Plugins/SourceToolbar` for this section. There we are
951 adding a toolbar with two buttons to create a sphere and a cylinder source. For
952 adding a toolbar, one needs to implement a subclass for
953 [`QActionGroup`][QActionGroup] which adds the [`QAction`][QAction]s for each of
954 the toolbar button and then implements the handler for the callback when the
955 user clicks any of the buttons. In the example `SourceToobarActions.{h,cxx}` is
956 the `QActionGroup` subclass that adds the two tool buttons.
957 
958 To build the plugin, the `CMakeLists.txt` file is:
959 
960 ```cmake
961 # This is a macro for adding QActionGroup subclasses automatically as toolbars.
962 paraview_plugin_add_action_group(
963  CLASS_NAME SourceToolbarActions
964  GROUP_NAME "ToolBar/SourceToolbar"
965  INTERFACES interfaces
966  SOURCES sources)
967 
968 # Now create a plugin for the toolbar. Here we pass the `interfaces` and
969 # `sources` returned by the above call.
970 paraview_add_plugin(SourceToolbar
971  VERSION "1.0"
972  UI_INTERFACES ${interfaces}
973  SOURCES ${sources}
974  SourceToolbarActions.cxx)
975 ```
976 
977 For the `GROUP_NAME`, we are using `ToolBar/SourceToolbar`; here `ToolBar` is a
978 keyword which implies that the action group is a toolbar (and shows up under
979 *View > Toolbars* menu) with the name `SourceToolbar`. When the plugin is
980 loaded, this toolbar will show up with two buttons.
981 
982 #### Adding a Menu
983 
984 Adding a menu to the menu bar of the main window is almost identical to adding
985 a toolbar. The only difference is that you use the keyword `MenuBar` in lieu
986 of `ToolBar` in the `GROUP_NAME` of the action group. So if you change the
987 `paraview_plugin_add_action_group` command above to the following, the plugin
988 will add a menu titled `MyActions` to the menu bar.
989 
990 ```cmake
991 paraview_plugin_add_action_group(
992  CLASS_NAME SourceToolbarActions
993  GROUP_NAME "MenuBar/MyActions"
994  INTERFACES interfaces
995  SOURCES sources)
996 ```
997 
998 If you give the name of an existing menu, then the commands will be added to
999 that menu rather than create a new one. So, for example, if the `GROUP_NAME`
1000 is `MenuBar/File`, the commands will be added to the bottom of the *File* menu.

1002 #### Autostart Plugins
1003 
1004 This refers to a plugin which needs to be notified when ParaView starts up or
1005 the plugin is loaded which ever happens later and then notified when ParaView
1006 quits. Example is in `Examples/Plugins/Autostart` in the ParaView source. For
1007 such a plugin, we need to provide a `QObject` subclass
1008 (`pqMyApplicationStarter`) with methods that need to be called on startup and
1009 shutdown.
1010 
1011 ```cpp
1012 class pqMyApplicationStarter : public QObject
1013 {
1014 Q_OBJECT
1015 public:
1016  // Callback for startup.
1017  // This cannot take any arguments
1018  void onStartup();
1019 
1020  // Callback for shutdown.
1021  // This cannot take any arguments
1022  void onShutdown();
1023 };
1024 ```
1025 
1026 The `CMakeLists.txt` looks as follows:
1027 
1028 ```cmake
1029 # Macro for auto-start plugins. We specify the class name and the methods to
1030 # call on startup and shutdown on an instance of that class. It returns the
1031 # interface and sources created in the variables passed to the `INTERFACES` and
1032 # `SOURCES` arguments, respectively.
1033 paraview_plugin_add_auto_start(
1034  CLASS_NAME pqMyApplicationStarter # the class name for our class
1035  STARTUP onStartup # specify the method to call on startup
1036  SHUTDOWN onShutdown # specify the method to call on shutdown
1037  INTERFACES interfaces
1038  SOURCES sources)
1039 
1040 # Create a plugin for this starter
1041 paraview_add_plugin(Autostart
1042  VERSION "1.0"
1043  UI_INTERFACES ${interfaces}
1044  SOURCES pqMyApplicationStarter.cxx ${interfaces})
1045 ```

1047 #### Adding new Representations for 3D View using Plugins
1048 
1049 ParaView's 3D view the most commonly used view for showing polygonal or
1050 volumetric data. By default, ParaView provides representation-types for showing
1051 the dataset as surface, wireframe, points etc. It’s possible to add
1052 representations using plugins that extends this set of available
1053 representation types.
1054 
1055 Before we start looking at how to write such a plugin, we need to gain some
1056 understanding of the 3D view and its representations. The 3D view uses 3 basic
1057 representation proxies for rendering all types of data:
1058 
1059  * (representations, `UnstructuredGridRepresentation`) – for
1060  `vtkUnstructuredGrid` or a composite dataset consisting of
1061  `vtkUnstructuredGrid`.
1062  * (representations, `UniformGridRepresentation`) – for `vtkImageData` or a
1063  composite dataset consisting of `vtkImageData`
1064  * (representations, `GeometryRepresentation`) – for all other data types.
1065 
1066 Each of these representation proxies are basically composite representation
1067 proxies that use other representation proxies to do the actual rendering, e.g.,
1068 `GeometryRepresentation` uses `SurfaceRepresentation` for rendering the data as
1069 wireframe, points, surface, and surface-with-edges and `OutlineRepresentation`
1070 for rendering an outline for the data. Subsequently, the 3 composite
1071 representation proxies provide a property named `Representation` which allows
1072 the user to pick the representation type he wants to see the data as. The
1073 composite representation proxy has logic to enable one of its internal
1074 representations based on the type chosen by the user.
1075 
1076 These 3 composite representation types are fixed and cannot be changed by
1077 plugins. What plugins can do is add more internal representations to any of
1078 these 3 composite representations to support new representations types that the
1079 user can choose using the representation type combo box on the display tab or
1080 in the toolbar.
1081 
1082 ![Representation type combo-box allowing user to choose the sub-representation to use](images/Representationplugin.png)
1083 
1084 ##### Using a New Mapper
1085 
1086 In this example, we see how to integrate a special polydata mapper written in
1087 VTK into ParaView. Let’s say the mapper is called `vtkMySpecialPolyDataMapper`
1088 which is simply a subclass of `vtkPainterPolyDataMapper`. In practice,
1089 `vtkMySpecialPolyDataMapper` can internally use different painters to do
1090 perform special rendering tasks.
1091 
1092 To integrate this mapper into ParaView, first we need to create a
1093 `vtkSMRepresentationProxy` subclass for that uses this mapper. In this example,
1094 since the mapper is a simple replacement for the standard
1095 `vtkPainterPolyDataMapper`, we can define our representation proxy as a
1096 specialization of the `SurfaceRepresentation` as follows:
1097 
1098 ```xml
1099 <ServerManagerConfiguration>
1100  <ProxyGroup name="representations">
1101  <RepresentationProxy
1102  name="MySpecialRepresentation"
1103  class="vtkMySpecialRepresentation"
1104  processes="client|renderserver|dataserver"
1105  base_proxygroup="representations"
1106  base_proxyname="SurfaceRepresentation">
1107  <Documentation>
1108  This is the new representation type we are adding. This is identical to
1109  the SurfaceRepresentation except that we are overriding the mapper with
1110  our mapper.
1111  </Documentation>
1112  </RepresentationProxy>
1113  </ProxyGroup>
1114 </ServerManagerConfiguration>
1115 ```
1116 
1117 `vtkMySpecialRepresentation` is a subclass of
1118 `vtkGeometryRepresentationWithFaces` where in the constructor we simply
1119 override the mappers as follows:
1120 
1121 ```cpp
1122 vtkMySpecialRepresentation::vtkMySpecialRepresentation()
1123 {
1124  // Replace the mappers created by the superclass.
1125  this->Mapper->Delete();
1126  this->LODMapper->Delete();
1127 
1128  this->Mapper = vtkMySpecialPolyDataMapper::New();
1129  this->LODMapper = vtkMySpecialPolyDataMapper::New();
1130 
1131  // Since we replaced the mappers, we need to call SetupDefaults() to ensure
1132  // the pipelines are setup correctly.
1133  this->SetupDefaults();
1134 }
1135 ```
1136 
1137 Next we need to register this new type with the any (or all) of the 3 standard
1138 composite representations so that it will become available to the user to
1139 choose in the representation type combo box. To decide which of the 3 composite
1140 representations we want to add our representation to, think of the input data
1141 types our representation supports. If it can support any type of data set, then
1142 we can add our representation all the 3 representations (as is the case with
1143 this example). However if we are adding a representation for volume rendering
1144 of `vtkUnstructuredGrid` then we will add it only to the
1145 `UnstructuredGridRepresentation`. This is done by using the `Extension` XML
1146 tag. It simply means that we are extending the original XML for the proxy
1147 definition with the specified additions. Now to make this representation
1148 available as a type to the user, we use the `RepresentationType` element , with
1149 `text` used as the text shown for the type in the combo-box, `subproxy`
1150 specifies the name of representation subproxy to activate when the user chooses
1151 the specified type. Optionally one can also specify the `subtype` attribute,
1152 which if present is the value set on a property named `Representation` for the
1153 subproxy when the type is chosen. This allows for the subproxy to provide more
1154 than one representation type.
1155 
1156 ```xml
1157 <ServerManagerConfiguration>
1158  <ProxyGroup name="representations">
1159  <Extension name="GeometryRepresentation">
1160  <Documentation>
1161  Extends standard GeometryRepresentation by adding
1162  MySpecialRepresentation as a new type of representation.
1163  </Documentation>
1164 
1165  <!-- this adds to what is already defined in PVRepresentationBase -->
1166  <RepresentationType
1167  subproxy="MySpecialRepresentation"
1168  text="Special Mapper"
1169  subtype="1" />
1170 
1171  <SubProxy>
1172  <Proxy name="MySpecialRepresentation"
1173  proxygroup="representations"
1174  proxyname="MySpecialRepresentation">
1175  </Proxy>
1176  <ShareProperties subproxy="SurfaceRepresentation">
1177  <Exception name="Input" />
1178  <Exception name="Visibility" />
1179  <Exception name="Representation" />
1180  </ShareProperties>
1181  </SubProxy>
1182  </Extension>
1183  </ProxyGroup>
1184 </ServerManagerConfiguration>
1185 ```
1186 
1187 The `CMakeLists.txt` file is not much different from what it would be like for
1188 adding a simple filter or a reader where the representation class is placed
1189 into the contained module.

1191 Source code for this example is available under
1192 `Examples/Plugins/Representation` in the ParaView source directory.

1200 
1201 There are several plugins that are included in ParaView source itself and are
1202 built as part of ParaView's build process. To add such a plugin to the ParaView
1203 build there are two options, adding it to the `ParaView/Plugins` directory is
1204 currently the only supported mechanism.
1205

1219 built executables.

1221 In your plugin source directory, ParaView searches for a file name
1222 `paraview.plugin` which provides ParaView with information about the plugin.

1224

1226 # Comments are allowed.

1231  build this plugin.

1233  # List of VTK modules required by the code contained in the plugin. This
1234  # allows ParaView to build the full set of requested modules if the plugin is

1237 ```

1240 `PARAVIEW_PLUGIN_ENABLE_PluginName` CMake option, then CMake will look for a
1241 `CMakeLists.txt` file next to the `paraview.plugin`. This file contains the
1242 calls to build the plugin including the `paraview_add_plugin` call, and
1243 building of any other libraries that the plugin needs.

1248 ## Plugins in Static Applications
1249 
1250 It is possible to import plugins into a ParaView-based application at compile
1251 time. When building ParaView-based applications statically, this is the only
1252 option to bring in components from plugins. When built statically (i.e., with
1253 `BUILD_SHARED_LIBS` set to false), ParaView will automatically link and load
1254 plugins that were enabled via CMake by inserting the necessary
1255 `PV_PLUGIN_IMPORT_INIT` and `PV_PLUGIN_IMPORT` macros.
1256 
1257 The code below shows how the `PV_PLUGIN` macros would be used to statically load
1258 plugins in custom applications:
1259 
1260 ```cpp
1261 #define PARAVIEW_BUILDING_PLUGIN
1262 #include "vtkPVPlugin.h"
1263 
1264 // Adds required forward declarations.
1265 PV_PLUGIN_IMPORT_INIT(MyFilterPlugin)
1266 PV_PLUGIN_IMPORT_INIT(MyReaderPlugin)
1267 
1268 class MyMainWindow : public QMainWindow
1269 {
1270  // ....
1271 };
1272 
1273 MyMainWindow::MyMainWindow(...)
1274 {
1275  // ... after initialization ...
1276 
1277  // Calls relevant callbacks to load the plugins and update the
1278  // GUI/Server-Manager
1279  PV_PLUGIN_IMPORT(MyFilterPlugin);
1280  PV_PLUGIN_IMPORT(MyReaderPlugin);
1281 }
1282 ```

1284 ## Pitfalls
1285 
1286 ### *Tools > Manage Plugins* is not visible!
1287 
1288 Plugins can only be loaded dynamically when ParaView is built with shared
1289 libraries. You must recompile ParaView with `BUILD_SHARED_LIBS=ON`.
1290 
1291 ### Compile error `invalid conversion from 'vtkYourFiltersSuperClass*' to 'vtkYourFilter*'`
1292 
1293 Any VTK object that needs to be treated as a filter or source has to be a
1294 `vtkAlgorithm` subclass. The particular superclass a filter is derived from has
1295 to be given not only in the standard C++ way:
1296 
1297 ```cpp
1298 class VTKMODULE_EXPORT vtkMyElevationFilter : public vtkElevationFilter
1299 ```
1300 
1301 but additionally declared with help of the `vtkTypeMacro`. For the example
1302 given above:
1303 
1304 ```cpp
1305 class VTKMODULE_EXPORT vtkMyElevationFilter : public vtkElevationFilter
1306 {

1326 ### Mysterious Segmentation Faults in Plugins that use Custom VTK Classes
1327 
1328 This primarily concerns plugins that make calls to your own custom `vtkMy` (or
1329 whatever you called it) library of VTK extensions.
1330 
1331 Symptoms:
1332 
1333  * The plugin will load, but causes a segfault when you try to use it.
1334  * If you use a debugger you may notice that in some cases when your code
1335  calls `vtkClassA.MethodB`, what actually gets called is
1336  `vtkClassC.MethodD`, where `MethodB` is a virtual member function. This is
1337  occurs because of different vtable entries in the Paraview-internal
1338  versions of the VTK libraries.
1339 
1340 The solution is to make sure that your `vtkMy` library is compiled against
1341 ParaView's internal VTK libraries. Even if you compiled VTK and ParaView using
1342 the same VTK sources, you *must not* link against the external VTK libraries.
1343 (The linker won't complain, because it will find all the symbols it needs, but
1344 this leads to unexpected behaviour.)
1345 
1346 To be explicit, when compiling your `vtkMy` library, you must set the CMake
1347 variable `VTK_DIR` to point to the `VTK` subdirectory in the directory in which
1348 you built ParaView. (On my system, CMake automatically finds VTK at
1349 `/usr/lib/vtk-5.2`, and I must change `VTK_DIR` to
1350 `~/source/ParaView3/build/VTK`.)

1352 ### "Is not a valid Qt plugin" in Windows
1353

1355 in doubt, try placing your plugin and all its dependent DLLs in the `bin`
1356 directory of your build and load it from there.
