1 Major API Changes {#MajorAPIChanges}
2 =================
3 
4 This page documents major API/design changes between different versions since we
5 started tracking these (starting after version 4.2).
6 
7 Changes in 5.9
8 --------------
9 
10 ###Representations and Ordered Compositing for Render View###
11 
12 We have refactored the code that manages how data redistribution is done when
13 ordered compositing is needed. Ordered compositing is used for rendering
14 translucent geometries or volume rendering in parallel. Previous implementation
15 used `vtkDistributedDataFilter` internally which required building of
16 `vtkPKdTree` for deciding how the data is distributed. The new implementation
17 uses `vtkRedistributeDataSetFilter` which supports arbitrary non-intersecting
18 bounding boxes. This also results is lots of code simplification. However, it
19 has resulted in API changes that will impact vtkPVDataRepresentation subclasses.
20 
21 * `vtkPVRenderView::MarkAsRedistributable` and
22  `vtkPVRenderView::SetOrderedCompositingInformation` have been removed. Instead one
23  should use `vtkPVRenderView::SetOrderedCompositingConfiguration` which enables
24  developers to specify exactly how the representation's data products
25  participate in the data-redistribution stage.
26 * For representations based on `vtkImageVolumeRepresentation`, such
27  representation no longer need to provide a `vtkExtentTranslator` subclass to
28  the render view via `vtkPVRenderView::SetOrderedCompositingInformation` to
29  enable the view to rebuild a KdTree. The view can now simply use the data
30  bounds provided either via `vtkPVRenderView::SetGeometryBounds` or
31  `vtkPVRenderView::SetOrderedCompositingInformation`.
32 
33 Since most custom representations are based on vtkImageVolumeRepresentation,
34 vtkUnstructuredGridVolumeRepresentation or vtkGeometryRepresentation, developers
35 are advised to look at the changes to those representations to get a better feel
36 for how to use `vtkPVRenderView::SetOrderedCompositingInformation` instead of
37 the legacy APIs.
38 
39 ###Changes where ParaView applications settings files are stored on Windows
40 
41 Due to a bug prior to 5.9 on Windows only, user settings JSON files (`<application>-UserSettings.json`)
42 written from ParaView-based applications were stored in the wrong directory on Windows.
43 Settings file are normally stored in a directory with the name of the organization,
44 but ParaView was storing them in a directory with the name of the application instead.
45 In ParaView 5.9, the `<application>-UserSettings.json` file is stored in the directory named
46 for the organization as expected.
47 
48 ###Run-time selection of OpenGL Widget rendering implementation###
49 
50 We have enabled run-time selection of Widget rendering OpenGL implementation, by
51 default it is set to _native_ mode. However, this can be changed to _Stereo_ mode
52 in startup time by adding the flag `--stereo` to the _Paraview_ binary.
53 
54 This also makes `pqQVTKWidget` not to extend from `pqQVTKWidgetBase`, instead,
55 extend from QWidget and contain (composition) either an instance of
56 `QVTKOpenGLNativeWidget` or `QVTKOpenGLStereoWidget`.
57 
58 ###Consistent font size in text source
59 
60 Corrected a setting that resulted in unwanted scaling of the font size from the
61 *Text Source* that scaled according to the size of the *RenderView*.

63 Changes in 5.8
64 ---------------
65 
66 ###Changes to caching in representation###
67 
68 Previously, caching for animation playback was handled by representations using
69 `vtkPVCacheKeeper` filter in the representation pipelines. This had several
70 issues:
71 
72 1. Each representations had to explicitly handle it, which was error-prone and
73  often ended up not being used correctly.
74 2. Caching happened on the pre-data-movement i.e. during animation playback, the
75  geometry still needed to be delivered to the rendering nodes.
76 3. Cache was incapable of distinguishing between pipelines that were modified
77  during animation playback and those that weren't. Consequently, animations
78  that simply moved the camera still ended up executing the representation
79  pipelines for each time step.
80 4. Not all views / representations participated in caching, only render view
81  did. Thus, if you had an animation that used a Spreadsheet view, for example,
82  you'd get no benefit from caching.
83 5. Even with caching enabled, animation playback was slow.
84 
85 
86 With 5.8, we have refactored handling of caches in views and representations.
87 Major highlights/changes are as follows:
88 
89 1. Caching is now the responsibility of the view. vtkPVDataDeliveryManager
90  handles caching along with data movement for views. All views, not just
91  render views, use vtkPVDataDeliveryManager (or subclass) to handle data
92  movement from data-processing nodes to rendering nodes. Representations
93  like `vtkChartRepresentation`, `vtkSpreadSheetRepresentation` can no longer
94  do data movement in `RequestData` but instead have to follow the same pattern
95  as `vtkGeometryRepresentation` and other render view representations to let
96  the view (using vtkPVDataDeliveryManager) handle data movement.
97  Consequently, if your `vtkPVDataRepresentation` subclass used a
98  `vtkPVCacheKeeper` in your representation pipeline, you can simply remove it.
99  Representation provide prepared data to the view using `vtkPVView::SetPiece`
100  in `vtkPVView::REQUEST_UPDATE` pass, and obtain delivered data for rendering
101  using `vtkPVView::GetDeliveredPiece` in `vtkPVView::REQUEST_RENDER`
102  (`vtkPVRenderView::GetPieceProducer` is still supported, but we encourage developers
103  to prefer the new `GetDeliveredPiece` API instead).
104 2. To ensure that the view creates a delivery manager of appropriate type, you
105  may have to add the delivery manager as a subproxy, as shown in the example
106  below:
107 
108  <SubProxy command="SetDeliveryManager">
109  <Proxy name="DeliveryManager"
110  proxygroup="delivery_managers"
111  proxyname="ContextViewDeliveryManager"/>
112  </SubProxy>
113 
114 3. When cached data is available, `REQUEST_UPDATE` request will no longer result
115  in a call to `vtkPVDataRepresentation::RequestData`. The representation will
116  be provided the cached data during `REQUEST_RENDER` pass. Note, if using
117  cache, you may get data that has older `MTime`, thus representation should
118  not assume monotonically increasing m-times for delivered data. Rendering
119  pipelines in representations can thus rely on the mtime and data pointer to
120  build their own rendering objects cache, if needed.
121 4. Representations always have to let the view do data transfer and obtain the
122  data to render from the view and never simply connect the data prepared in
123  `RequestData` to mappers, for example. This was acceptable in certain cases
124  before, but not longer supported since `RequestData` will not get called
125  when cache data is being used.

127 ###Plugin loading (including auto loaded plugins)###
128 
129 With this version, ParaView is now formalizing that all plugins, including the
130 ones that are flagged as "auto-load" are loaded **after** the main window has
131 been instantiated. This ensures that plugin developers can assume existence of a
132 main window irrespective of whether the plugin was auto-loaded or loaded after
133 the application has started.

136 ###GlobalPropertyLink XML hints###
137 
138 The implementation for settings proxies and the proxy for color palette has been
139 combined thus removing the need for a separate `vtkSMGlobalPropertiesProxy`
140 class. The color-palette proxy is now same as other settings proxies and is
141 registered under `settings` group and not `global_properties` group. To simply
142 XML hints to reflect this change, we have deprecated `GlobalPropertyLink` hint.
143 Following XML snippet shows how to change existing `GlobalPropertyLink` element
144 to `PropertyLink`.
145 
146  <!-- OLD XML -->
147  <DoubleVectorProperty name="GridColor"
148  number_of_elements="3" ... >
149  ...
150  <Hints>
151  <GlobalPropertyLink type="ColorPalette" property="ForegroundColor" />
152  </Hints>
153  </DoubleVectorProperty>
154 
155  <!-- NEW XML -->
156  <DoubleVectorProperty name="GridColor"
157  number_of_elements="3" ... >
158  ...
159  <Hints>
160  <PropertyLink group="settings" proxy="ColorPalette" property="ForegroundColor"
161  unlink_if_modified="1" />
162  </Hints>
163  </DoubleVectorProperty>
164 
165 Note, the `PropertyLink` hint was already present and used in earlier versions
166 for linking with properties on other settings proxies. We simply are following
167 the same pattern for color palette. The new `unlink_if_modified` attribute
168 ensures that the link with the settings proxy is broken if the user explicitly
169 modifies the property.

171 Changes in 5.7

174 ###vtkSMFieldDataDomain in array selection properties###
175 
176 Previously, `vtkSMFieldDataDomain` was added to `vtkSMStringVectorProperty`
177 instances that allowed user to choose the array to process. This is no longer needed
178 or supported. Simply remove the `vtkSMFieldDataDomain` from the XML for such properties.
179 Typically such properties have `vtkSMArrayListDomain` and that domain alone is
180 sufficient to enable selection of array name and its association.
181 
182 ###Changes to vtkSMFieldDataDomain###
183 
184 vtkSMFieldDataDomain no longer updates the domain based on the type of the
185 input. The domain will always list the attribute types known to VTK/ParaView. As
186 a result, XML attributes `force_point_cell_data` and
187 `disable_update_domain_entries` for the XML definition of vtkSMFieldDataDomain
188 have been deprecated and should simply be removed from the XML as they are no
189 longer relevant.

191 ##pqDoubleLineEdit API changes##
192 
193 `pqDoubleLineEdit` now is simply a `pqLineEdit` with no signficant API differences
194 as far as setting and getting text values is concerned. This implies, however, that
195 tests that were updated in ParaView 5.6 to use `set_full_precision_text` instead of
196 `set_string` must now be reverted back to use `set_string`.
197 
198 The following command was adequate for ParaView test changes:
199 `git grep -l set_full_ | xargs sed -i 's/set_full_precision_text/set_string/g'`.
200 
201 ##Views and Layouts##
202 
203 Previously, when a new view is created ParaView automatically associated it with
204 a layout. This is no longer the case. Associating a view with a layout is now an
205 explicit action.
206 `vtkSMParaViewPipelineControllerWithRendering::AssignViewToLayout` may be used
207 to help assign a view to an available layout (or create a new one, if none
208 exists). This was essential for addressing several layout related issues such
209 as paraview/paraview#18964, and paraview/paraview#18963.

238 ###Changes to Python shell###
239 
240 `pqPythonShell` now is a fairly self contained class. One no longer has to go
241 through `pqPythonManager` or `pqPythonDialog` to create it. `pqPythonDialog` has
242 been deprecated. `pqPythonManager` has been relegated to a helper class to

248 supported.

250 ###Changes to offscreen rendering options###
251 
252 ParaView executables now automatically choose to create on-screen or off-screen
253 render windows for rendering based on the executable type and current
254 configuration. As a result **"UseOffscreenRendering"** property on views has
255 been removed to avoid conflicting with the new approach. Executables have new
256 command line arguments: `--force-offscreen-rendering` and
257 `--force-onscreen-rendering` that can be used to override the default behavior
258 for the process.

260 ###Changes to vtkArrayCalculator
261 `vtkArrayCalculator::SetAttributeMode` is deprecated in favor of `vtkArrayCalculator::SetAttributeType`
262 which takes vtkDataObject attribute modes instead of custom constants as its parameter value.
263 The ParaView calculator filter's AttributeMode property's values changed as a result of this.
264 
265 ###Changes to vtkSMViewProxy::CaptureWindowInternal###
266 
267 `vtkSMViewProxy::CaptureWindowInternal` now takes a 2-component magnification
268 factor rather than a single component. That allows for more accurate target image
269 resolution than before (see #17567).
270 
271 ###Replaced LockScalarRange property in PVLookupTable with AutomaticRescaleRangeMode###
272 
273 The `LockScalarRange` property in the `PVLookupTable` proxy has been removed. It
274 has been replaced by the `AutomaticRescaleRangeMode` property. When
275 `AutomaticRescaleRangeMode` is set to `vtkSMTransferFunctionManager::Never`,
276 the transfer function minimum and maximum value is never updated, no matter what
277 event occurs. This corresponds to when `LockScalarRange` was set to "on" in
278 previous versions of ParaView. When `AutomaticRescaleRangeMode` is set to a
279 different option, that option governs how and when the transfer function is
280 reset. This option overrides whichever option is set in the `GeneralSettings`
281 property `TransferFunctionResetMode`.
282 
283 The `TransferFunctionResetMode` property still exists, but it has been slightly
284 repurposed. No longer does it control the range reset mode for ALL transfer
285 functions. Instead, it serves the following functions:
286 
287 * determining the `AutomaticRescaleRangeMode` when a lookup table is created, and
288 
289 * determining what the `AutomaticRescaleRangeMode` should be when the data array
290  range is updated to the data range of the selected representation
291 
292 As part of this change, the enum values
293 
294  GROW_ON_APPLY
295  GROW_ON_APPLY_AND_TIMESTEP
296  RESET_ON_APPLY
297  RESET_ON_APPLY_AND_TIMESTEP
298 
299 have been moved from `vtkPVGeneralSettings.h` to `vtkSMTransferFunctionManager.h`.

369 ###Qt 5 Support (replacing QVTKWidget)###
370 
371 ParaView has switched over to using Qt 5 by default. While for most cases the
372 application code should be unaffected, if your app was directly creating (or
373 using) `QVTKWidget` however, you will need to change it. When
374 building with Qt 5, you are expected to use `QVTKOpenGLWidget` instead of
375 `QVTKWidget` (which is still needed when building with Qt 4).
376 Since ParaView supports building with Qt 4 and 5 for this release, to make it
377 easier, we have the following solution.
378 
379 A new header **pqQVTKWidgetBase.h** defines a new typedef `pqQVTKWidgetBase`.
380 Include this header in your code and replace `QVTKWidget` with `pqQVTKWidgetBase`.
381 `pqQVTKWidgetBase` gets typedef'ed to `QVTKWidget` for Qt 4 builds and to
382 `QVTKOpenGLWidget` for Qt 5 builds.
383 
384 If your existing code was creating and setting up a new instance of
385 `QVTKWidget`, you need to change it further. QVTKOpenGLWidget cannot work with
386 any `vtkRenderWindow` (as was the case with QVTKWidget),
387 it instead needs a `vtkGenericRenderWindow` (see the documentation
388 for `QVTKOpenGLWidget` for details). To make that easier, **pqQVTKWidgetBase.h**
389 defines another typedef `pqQVTKWidgetBaseRenderWindowType`. The code to create
390 and setup QVTKWidget can then be changed as follows:
391 
392  vtkNew<pqQVTKWidgetBaseRenderWindowType> renWindow;
393  pqQVTKWidgetBase* widget = new pqQVTKWidgetBase(...);
394  widget->SetRenderWindow(renWindow.Get());
395 
396 If you are using a `vtkView` to create the viewport. You can set that up as
397 follows:
398 
399  vtkNew<pqQVTKWidgetBaseRenderWindowType> renWindow;
400  pqQVTKWidgetBase* widget = new pqQVTKWidgetBase(...);
401  widget->SetRenderWindow(renWindow.Get());
402 
403  vtkNew<vtkContextView> view;
404  view->SetRenderWindow(renWindow.Get());
405 
406 If your application supports Qt 5 alone, when you should directly use
407 `QVTKOpenGLWidget`. `pqQVTKWidgetBase` is a temporary solution to support
408 building with Qt 5 and Qt 4. Qt 4 support will be deprecated and remove in
409 future releases.

419 ###Removed pqStandardArrayColorMapsBehavior###
420 
421 Setting the default colormaps from a Qt Behavior was causing inconsistent results
422 between the ParaView UI and pvpython/pvbatch. Adding these default settings was
423 moved into the server manager. To override these defaults, use
424 vtkSMSettings::AddCollectionFromX with a priority greater than 0. The settings
425 affected are the default colormap for arrays with names vtkBlockColors and
426 AtomicNumbers. With these settings moved, the Qt behavior
427 pqStandardArrayColorMapsBehavior no longer does anything and so it has been removed.

429 ###Qt dependencies###
430 
431 Starting with 5.2, ParaView natively supports Qt 4 and Qt 5. To simplify writing
432 code with either dependency, we now have a new CMake file `ParaViewQt.cmake`
433 that gets included by `PARAVIEW_USE_FILE`. This provides new macros that be used
434 to find qt (`pv_find_package_qt`), wrap cpp (`pv_qt_wrap_cpp`), ui
435 (`pv_qt_wrap_ui`), or add resources (`pv_qt_add_resources`) instead of using
436 `qt4_` or `qt4_` specific versions based on whether the app is being built with
437 Qt4 or Qt5. `pv_find_package_qt` accepts optional `QT5_COMPONENTS` and
438 `QT4_COMPONENTS` which can be used to list the Qt component dependencies for
439 each of the versions. e.g.
440 
441  include(ParaViewQt) # generally not needed, since auto-included
442  pv_find_package_qt(qt_targets
443  QT4_COMPONENTS QtGui
444  QT5_COMPONENTS Widgets)
445 
446  pv_qt_wrap_cpp(moc_files ${headers})
447  pv_qt_wrap_ui(ui_files ${uis})
448 
449  ...
450  target_link_libraries(${target} LINK_PRIVATE ${qt_targets})

471 ###Removed Cube Axes###

481

484 created a new, more generic, pqProxyWidgetDialog that also allows the user to

488 
489 ###Refactored 3DWidget panels###
490 
491 3DWidget panels were subclasses of `pqObjectPanel`, the Properties panel
492 hierarchy that has been deprecated since 4.0. This version finally drops support
493 for these old 3D widget panels. The new implementation simply uses the
494 `pqPropertyWidget` infrastructure. A 3D widget panel is nothing more than a custom
495 `pqPropertyWidget` subclass for a property-group (`vtkSMPropertyGroup`) which
496 creates an interactive widget to be shown in the active view to help the users
497 change the values for properties in the property-group using this interactive
498 widget, in addition to standard Qt-based UI elements.
499 
500 If you have a filter or proxy indicating that the Properties panel use one of
501 the standard 3D widgets for controlling certain properties on that proxy, you
502 specified that using `<Hints>` in the Proxy's XML definition.
503 
504 For example, the **(implicit_functions, Plane)** proxy requested that
505 `pqImplicitPlaneWidget` be used using the following XML hints.
506 
507  <Proxy class="vtkPVPlane" name="Plane">
508  <InputProperty is_internal="1" name="Input" />
509  <DoubleVectorProperty command="SetOrigin"
510  default_values="0.0 0.0 0.0"
511  name="Origin"
512  number_of_elements="3">
513  ...
514  </DoubleVectorProperty>
515  <DoubleVectorProperty command="SetNormal"
516  default_values="1.0 0.0 0.0"
517  name="Normal"
518  number_of_elements="3">
519  ...
520  </DoubleVectorProperty>
521  <DoubleVectorProperty animateable="1"
522  command="SetOffset"
523  default_values="0.0"
524  name="Offset"
525  number_of_elements="1">
526  </DoubleVectorProperty>
527  <Hints>
528  <PropertyGroup type="Plane">
529  <Property function="Origin" name="Origin" />
530  <Property function="Normal" name="Normal" />
531  </PropertyGroup>
532  <ProxyList>
533  <Link name="Input" with_property="Input" />
534  </ProxyList>
535  </Hints>
536  </Proxy>
537 
538 Since 3D widget panels are now simply custom `pqPropertyWidget` subclasses for a
539 property-group, this code changes are follows:
540 
541  <Proxy class="vtkPVPlane" name="Plane">
542  <InputProperty is_internal="1" name="Input" />
543  <DoubleVectorProperty command="SetOrigin"
544  default_values="0.0 0.0 0.0"
545  name="Origin"
546  number_of_elements="3">
547  ...
548  </DoubleVectorProperty>
549  <DoubleVectorProperty command="SetNormal"
550  default_values="1.0 0.0 0.0"
551  name="Normal"
552  number_of_elements="3">
553  ...
554  </DoubleVectorProperty>
555  <DoubleVectorProperty animateable="1"
556  command="SetOffset"
557  default_values="0.0"
558  name="Offset"
559  number_of_elements="1">
560  </DoubleVectorProperty>
561  <PropertyGroup label="Plane Parameters" panel_widget="InteractivePlane">
562  <Property function="Origin" name="Origin" />
563  <Property function="Normal" name="Normal" />
564  <Property function="Input" name="Input" />
565  </PropertyGroup>

620 ###Changes to vtkSMInputArrayDomain###
621 
622 vtkSMInputArrayDomain has changed the meaning of **"any"** attribute type. It
623 now includes field data arrays. To exclude field data arrays from the field
624 selection, change this to **"any-except-field"**. This is also the default for
625 vtkSMInputArrayDomain, hence simply removing the `attribute_type` field is also
626 an acceptable solution and is the recommended approach so that the XML can work

629 Changes in 5.0
630 --------------
631 
632 ###Changed *Source* property to **GlyphType** on certain representations###
633 
634 *Source* property on representations, which was not exposed in the UI has been
635 changed to *GlyphType* to better match its role. Since this property was not
636 exposed earlier, this change should not affect any users except custom
637 applications that explicitly used this property.
638 
639 
640 ###Changes to caching infrastructure in **vtkPVDataRepresentation** and subclasses###
641 
642 To avoid extra work and sync issues when propagating **UseCache** and
643 **CacheKey** flags from a vtkPVView to vtkPVDataRepresentation,
644 vtkPVDataRepresentation was updated to directly obtain thet values for these
645 flags from the view. To enable this, vtkPVDataRepresentation saves a weak
646 reference to the View it's being added to in vtkPVDataRepresentation::AddToView
647 and vtkPVDataRepresentation::RemoveFromView. Subclasses of
648 vtkPVDataRepresentation didn't consistently call the superclass implementation,
649 hence any representation subclass should ensure that it calls the superclass
650 implementations of AddToView and RemoveFromView for this to work correctly.
651 vtkPVView::AddRepresentationInternal checks for this and will raise a runtime
652 error if a representaiton is encountered that didn't properly set its View.
653 
654 The above change also makes **vtkPVDataRepresentation::SetUseCache** and
655 **vtkPVDataRepresentation::SetCacheKey** obsolete. Subclasses no longer need to
656 provide any implementation for these methods and they should simply be removed.

658 Changes in 4.4
659 --------------
660 
661 ###Refactored pqView widget creation mechanisms###
662 
663 pqView now adds a pure virtual method pqView::createWidget(). Subclasses should
664 override that method to create the QWidget in which the view is *rendered*. In
665 the past the subclasses implemented their own mechanisms to create the widget
666 and return it when `getWidget()` was called the first time.
667 
668 `pqView::getWidget()` is now deprecated. Users should use pqView::widget()
669 instead. This method internally calls the pqView::createWidget() when
670 appropriate.
671 
672 ###Removed vtkPVGenericRenderWindowInteractor, vtkPVRenderViewProxy###
673 
674 ParaView was subclassing vtkRenderWindowInteractor to create
675 `vtkPVGenericRenderWindowInteractor` to handle interaction. That piece of code
676 was potentially derived from an older implementation of
677 vtkRenderWindowInteractor and hence did what it did. Current implementation of
678 vtkRenderWindowInteractor lets the vtkInteractionStyle (and subclasses) do all
679 the heavy lifting. ParaView did that to some extent (since it has a
680 vtkPVInteractorStyle), but will was relying on
681 `vtkPVGenericRenderWindowInteractor`, `vtkPVRenderViewProxy` to propagate
682 interaction/still renders and other things. This has been refactored. ParaView
683 no longer uses a special vtkRenderWindowInteractor. All logic is handled by
684 observers on the standard vtkRenderWindowInteractor.
685 
686 This change was done to make it easier to enable interactivity in `pvpython`.
687 
688 See also: vtkSMRenderViewProxy::SetupInteractor(). Subclasses of pqView now pass
689 the interactor created by QVTKWidget to this method to initialize it.
690 
691 See also: vtkSMViewProxyInteractorHelper.
692 
693 ###Refactored Color Preset Management (and UI)###
694 
695 The color preset management has been completely refactored for this release.
696 This makes presets accessible to Python clients.
697 `vtkSMTransferFunctionPresets` is the class that manages the presets in the
698 *ServerManager* layer.
699 
700 On the UI side, `pqColorPresetModel`, `pqColorPresetManager`, and
701 `pqColorMapModel` no longer exist. They have been replaced by `pqPresetDialog`.
702 Since the UI is completely redesigned, tests in custom applications
703 that used the color preset dialog will need to be updated as well.
704 
705 The preset themselves are now serialized as JSON in the same form as the
706 settings JSON. ColorMaps in legacy XML format are still loadable from the UI. At
707 the same time, a tool `vtkLegacyColorMapXMLToJSON` is available to convert such
708 XMLs to JSON.

710 ###Changes to `pqViewFrame`###
711 
712 Commit [afaf6a510](https://gitlab.kitware.com/paraview/paraview/-/commit/afaf6a510ecb872c49461cd850022817741e1558)
713 changes the internal widgets created in `pqViewFrame` to add a new `QFrame` named
714 **CentralWidgetFrame** around the rendering viewport. While this shouldn't break any

716 XML is fairly simple. Just add the **CentralWidgetFrame** to the widget hierarchy at the appropriate
717 location. See the original
718 [merge request](https://gitlab.kitware.com/paraview/paraview/-/merge_requests/167)
719 for details.

721 ###Changes to `vtkSMProxyIterator`###
722 
723 `vtkSMProxyIterator::Begin(const char* groupName)` now additionally
724 sets the iterator mode to iterate over one group. This will likely
725 break any code that uses this method, since the iterated data list will
726 change. To recover the original functionality of the iterator, simply call
727 `vtkSMProxyIterator::SetModeToAll()` after calling
728 `vtkSMProxyIterator::Begin(const char* groupName)`.

730 Changes in 4.3
731 --------------
732 
733 ###Replaced `pqCurrentTimeToolbar` with `pqAnimationTimeWidget`###
734 
735 `pqCurrentTimeToolbar` was a `QToolbar` subclass designed to be used in an
736 application to allow the users to view/change the animation time. We
737 reimplemented this class as `pqAnimationTimeWidget` making it a `QWidget`
738 instead of `QToolbar` to make it possible to use that in other widgets e.g.
739 `pqTimeInspectorWidget`. `pqAnimationTimeToolbar` was also changed to use
740 `pqAnimationTimeWidget` instead of `pqCurrentTimeToolbar`.
741 
742 Besides just being a cleaner implementation (since it uses `pqPropertyLinks`
743 based linking between ServerManager and UI), `pqAnimationTimeWidget` also allows
744 applications to change the animation *play mode*, if desired. The rest of the
745 behavior of this class is similar to `pqCurrentTimeToolbar` and hence should not
746 affect applications (besides need to update any tests since the widget names
747 have now changed).
748 
749 ###Removed the need for plugins to specify a GUI_RESOURCE_FILE###
750 
751 It is cleaner and easier to update one xml file for each plugin.
752 Hints in the server manager xml file already perform the same functionality and
753 the GUI xml file was deprecated. Sources are now added to the **Sources** menu
754 and filters are added to the **Filters** menu when they are detected in the server
755 xml file. To place sources/filters in a category, add the following hint to the
756 Proxy XML element (`category` and `icon` attributes are optional):
757 
758  <SourceProxy ...>
759  ...
760  <Hints>
761  <ShowInMenu category="<category-name>" icon="<qt-image-resource>" />
762  </Hints>
763  </SourceProxy>
764 
765 The ability to add filters to the sources menu (as done by a few plugins) is
766 removed by this change.
767 
768 This change also modifies how readers are detected. Readers now must provide a
769 hint to the ReaderManager in the server xml file to be detected as readers rather
770 than sources. The backwards compatibility behavior that assumed any source with
771 a FileName attribute was a reader has been removed.
772 
773 ###Removed pqActiveView (use pqActiveObjects instead)###
774 
775 `pqActiveView` was a long deprecated class which internally indeed used
776 pqActiveObjects. `pqActiveView` has now been removed. Any code using
777 `pqActiveView` can switch to using pqActiveObjects with very few code changes.
