1 ParaView and Offscreen Rendering {#Offscreen}
2 ================================
3 
4 ParaView is often used to render visualization results. While in most cases,
5 the results are presented to the users on their screens or monitors, there are
6 cases when this is not required. For example, if running a batch script to
7 generate several images, one may not care to see windows flashing in front of
8 them while the results are being generated. Another use-case is when using a
9 **pvserver** or **pvrenderserver** to render images remotely and having them
10 sent back to the client for viewing. The server-side windows are do not serve
11 any useful purpose in that mode. In other cases, showing a window may not be
12 possible at all, for example on certain Linux clusters, one may not have access
13 to an X server at all, in which case on-screen rendering is just not possible
14 (this is also referred to as headless operating mode).
15 
16 This page documents the onscreen, offscreen, and headless rendering support in
17 ParaView.
18 
19 ##Caveats##
20 
21 In this page, we limit ourselves to a Linux-based system. On Windows and macOS,
22 offscreen support is available, but true headless operation is currently not
23 widely used and hence left out of this discussion.
24 
25 ##Terminology##
26 
27 A brief explanation of the terms:
28 
29 * **Desktop Qt client**: This refers to the Qt-based GUI. This is launched using
30  the **paraview** executable.
31 
32 * **Command line executables**: This refers to all other ParaView executables,
33  e.g. **pvserver**, **pvdataserver**, **pvrenderserver**, **pvpython**,
34  and **pvbatch**.
35 
36 * **Onscreen**: Onscreen refers to the case where the rendering results are
37  presented to the users in a viewable window. On Linux, this invariably requires
38  a running and accessible X server. The desktop Qt client can only operate in
39  on-screen mode and hence needs an accessible X server.
40 
41 * **Offscreen**: Offscreen simply means that the rendering results are not presented
42  to the user in a window. On Linux, this does not automatically imply that an
43  accessible X server is not needed. X server may be needed, unless ParaView
44  was built with an OpenGL implementation that allows for headless operation.
45  This mode is not applicable to the desktop Qt client and is only supported
46  by the command line executables.
47 
48 * **Headless**: Headless rendering automatically implies offscreen rendering.
49  In addition, on Linux, it implies that the rendering does not require an
50  accessible X server nor will it make any X calls.
51 
52 **Onscreen** and **offscreen** support is built by default. Thus ParaView
53 binaries available from paraview.org support these modes. **Headless** support
54 requires special builds of ParaView with runtime capabilities that are not
55 widely available yet. Hence currently (as of ParaView 5.4) one has to build
56 ParaView from source with special build flags to enable headless support.
57 
58 ##OpenGL Implementations##
59 
60 ParaView uses OpenGL for rendering. OpenGL is an API specification for 2D/3D
61 rendering. Many vendors provide implementations of this API, including those
62 that build GPUs.
63 
64 For sake of simplicity, let's classify OpenGL implementations as **hardware
65 (H/W)** and **software (S/W)**. **H/W** includes OpenGL implementations provided by
66 NVIDIA, ATI, Intel, Apple and others which typically use the system hardware
67 infrastructure for rendering. The runtime libraries needed for these are
68 available on the system. **S/W** currently includes Mesa3D -- a software
69 implementation of the OpenGL standard. Despite the names, H/W doesn't
70 necessarily imply use of GPUs, nor does S/W imply exclusion of GPUs. Nonetheless
71 we use this naming scheme as it has been prevalent.
72 
73 ###APIs for Headless Support###
74 
75 Traditionally, OpenGL implementations are coupled with the window system to
76 provide an OpenGL context. Thus, they are designed for non-headless operation.
77 When it comes to headless operation, there are alternative APIs that an
78 application can use to create the OpenGL context that avoid this dependency on
79 the window system (or X server for the sake of this discussion).
80 
81 Currently, ParaView supports two distinct APIs that are available for headless
82 operation: **EGL** and **OSMesa** (also called **Offscreen Mesa**). It must be
83 noted that headless support is a rapidly evolving area and changes are expected
84 in coming months. Modern H/W OpenGL implementations support EGL while S/W
85 (or Mesa) supports OSMesa. One has to build ParaView with specific CMake flags
86 changed to enable either of these APIs. Which headless API you choose in your
87 build depends on which OpenGL implementation you plan to use.

89 ##ParaView Builds##
90 
91 Before we look at the various ways you can build and use ParaView, let's
92 summarize relevant CMake options available:
93 
94 * `VTK_USE_X`: When ON, implies that ParaView can link against X libraries.
95  This allows ParaView executables to create on-screen windows, if needed.
96 
97  When `VTK_USE_X` is ON, these variables must be specified:
98  * `OPENGL_INCLUDE_DIR`: Path to directory containing `GL/gl.h`.
99  * `OPENGL_gl_LIBRARY`: Path to `libGL.so`.
100  * `OpengL_glu_LIBRARY`: not needed for ParaView; leave empty.
101  * `OPENGL_xmesa_INCLUDE_DIR`: not needed for ParaView; leave empty.
102 
103 * `VTK_OPENGL_HAS_OSMESA`: When ON, implies that ParaView can use OSMesa
104  to support headless modes of operation.
105 
106  When `VTK_OPENGL_HAS_OSMESA` is ON, these variables must be specified:
107  * `OSMESA_INCLUDE_DIR`: Path to containing `GL/osmesa.h`.
108  * `OSMESA_LIBRARY`: Path to `libOSMesa.so`
109 
110 * `VTK_OPENGL_HAS_EGL`: When ON, implies that ParaView can use EGL to support
111  headless modes of operation.
112 
113  When `VTK_OPENGL_HAS_EGL` is ON, these variables must be specified:
114  * `EGL_INCLUDE_DIR`: Path to directory containing `GL/egl.h`.
115  * `EGL_LIBRARY`: Path to `libEGL.so`.
116  * `EGL_opengl_LIBRARY`: Path to `libOpenGL.so`.
117 
118 * `PARAVIEW_USE_QT` indicates if the desktop Qt client should be built.
119 
120 
121 All combinations of above options can be turned on or off independently except that
122 presently `VTK_OPENGL_HAS_EGL` and `VTK_OPENGL_HAS_OSMESA` are mutually exclusive i.e.
123 only one of the two can be ON at the same time. This is because the current version of Mesa (17.1.5)
124 doesn't support EGL for OpenGL, it's only supported for OpenGL-ES.
125 
126 A few things to note:
127 * If `VTK_OPENGL_HAS_EGL` or `VTK_OPENGL_HAS_OSMESA` is ON, the build supports headless rendering, otherwise
128  `VTK_USE_X` must be ON and the build does not support headless, but can still support offscreen rendering.
129 * If `VTK_USE_X` is OFF, then either `VTK_OPENGL_HAS_OSMESA` or `VTK_OPENGL_HAS_EGL` must be ON. Then the build
130  does not support onscreen rendering, but only headless rendering.
131 * If `PARAVIEW_USE_QT` is ON and `VTK_USE_X` is ON, while ParaView command line tools won't
132  link against or use X calls, Qt will and hence an accessible X server is still needed to run the
133  desktop client.
134 * If `VTK_OPENGL_HAS_OSMESA` is ON, and `VTK_USE_X` is ON, then all the OpenGL and OSMesa variables
135  should point to the Mesa libraries.
136 * Likewise, if `VTK_OPENGL_HAS_EGL` is ON and `VTK_USE_X` is ON, then all the OpenGL and EGL variables should point
137  to the system libraries providing both, typically the NVidia libraries.
138 
139 ##Default Rendering Modes##
140 
141 Since now it's possible to build ParaView with onscreen and headless support simultaneously, which type of render window the ParaView
142 executable creates by default also needs some explanation.
143 
144 * The ParaView desktop Qt client always creates an onscreen window using GLX calls via Qt.
145 * `pvserver`, `pvrenderserver` and `pvbatch` always create an offscreen render window. If built with headless support,
146  it will be an offscreen-headless window. There are a few exceptions where it will create an onscreen window instead:
147  - if running in tile-display mode, i.e. `-tdx` or `-tdy` command line options are passed
148  - if running in immersive mode e.g. CAVE
149  - if `PV_DEBUG_REMOTE_RENDERING` environment is set
150  - if `--force-onscreen-rendering` command line option is passed.
151 * `pvpython` always creates an onscreen render window if built with onscreen support. The following are the exceptions when it
152  creates offscreen (or headless if supported) render windows.
153  - if `--force-offscreen-rendering` command line option is passed.
154 
155 `--use-offscreen-rendering` command line option supported by ParaView 5.4 and earlier has now been deprecated and is
156 interpreted as `--force-offscreen-rendering`.
