1 Subset Inclusion Lattice (SIL) {#SubsetInclusionLattice}
2 ================
3 
4 Data is complex! As simulations have grown in complexity and capabilities, the
5 datasets they produce have also become complex. It is not uncommon to encounter
6 datasets comprising of multiple chunks or blocks e.g. Exodus, CGNS. The blocks may also
7 have relationships between them e.g. to describe an assembly, or block properties like
8 materials. When such relationships are available in the dataset, it is indeed
9 useful to present this information to the users so that they can choose to load
10 blocks in multiple ways. Subset inclusion lattice or SIL is the mechanism
11 available to ParaView readers to describe this information.
12 
13 Currently, SIL is only used by readers to describe relations among blocks to
14 enable selecting of blocks to reader. The blocks need not corresponding to
15 actual data blocks in the file but in practice, they often do. If future, the
16 plan is to provide access to the SIL along with composite dataset produced by
17 the reader.
18 
19 
20 Generating a SIL
21 ----------------
22 
23 A reader or a data source that wants to expose an ability to select blocks to
24 read should generate a SIL. While a SIL can include information of data arrays
25 such as point data arrays or cell data arrays available in the file, currently
26 we keep that separate due to existing conventions.
27 
28 In `vtkAlgorithm::RequestInformation`, every time the filename changes, you read
29 metadata from the file to build a SIL (vtkSubsetInclusionLattice).
30 vtkSubsetInclusionLattice provides API to initialize the SIL and then add nodes
31 with names in a tree structure with parent-child relationships. Directed links
32 between subtrees can also be added -- these are called cross links
33 (vtkSubsetInclusionLattice::AddCrossLink).
34 
35 The top-level nodes i.e. nodes with parent 0 are treated as classifications or
36 categories e.g. **Assembly**, **Material** etc. There are no other special requirements
37 on what the nodes should be or how the relationships are defined. Typically, the
38 first category subtree is native structure for the file format. For example, in
39 CGNS, data is laid out in a tree-like structure comprising of bases, zones, and
40 patches. Hence the CGNS reader (vtkCGNSReader) chose to put this structure under
41 the first **Hierarchy** subtree. The structure reflects the file format layout.
42 
43 
44  /Hierarchy/
45  Base1/
46  Zone1/
47  Grid
48  Patch0
49  Patch1
50  Zone2/
51  Grid
52  Patch2
53  Base2/
54  ...
55 
56 
57 The zones and patches have an attribute called *family name*. The family is
58 often used to describe additional characteristics of the zone or patch e.g
59 *boundary-condition-1*, *side-wall*, etc. To allow the ability to select using
60 family names, vtkCGNSReader added a second category subtree for **Families**.
61 
62  /Families/
63  Family1
64  Family2
65  ...
66 
67 To link family names with nodes under the Hierarchy sub-tree, we use cross
68 links. Thus there's a cross link from each family name to the zone, grid or patch nodes
69 under the Hierarchy sub-tree.
70

74  base/
75  blk-1/
76  Grid
77  bc-1-wall

79  bc-3-outflow

81  bc-5-symm-z1
82  bc-6-symm-z2
83  /Families/
84  bc-1-wall
85  (cross-link: Hierarchy/base/blk-1/bc-1-wall)
86  bc-2-inflow
87  (cross-link: Hierarchy/base/blk-1/bc-2-inflow)
88  bc-3-outflow
89  (cross-link: Hierarchy/base/blk-1/bc-3-outflow)
90  bc-4-symm-y
91  (cross-link: Hierarchy/base/blk-1/bc-4-symm-y)
92  bc-5-symm-z1
93  (cross-link: Hierarchy/base/blk-1/bc-5-symm-z1)

95  (cross-link: Hierarchy/base/blk-1/bc-6-symm-z2)
96  /Grids/
97  base/

99  (cross-link: Hierarchy/base/blk-1/Grid)
100  /Patches/
101  base/
102  blk-1
103  (cross-link: Hierarchy/base/blk-1/bc-1-wall)
104  (cross-link: Hierarchy/base/blk-1/bc-2-inflow)

109 
110 Now, when user selects a specific family, all nodes linked to it under the
111 Hierarchy subtree also need to be selected. vtkSubsetInclusionLattice manages
112 this internally.
113 
114 While vtkSubsetInclusionLattice has API to select/deselect nodes by node ids as
115 well as user-friendly paths, it's inconvenient for a CGNS user to know how a
116 base or family is to be named. Hence vtkCGNSReader defines a subclass of
117 vtkSubsetInclusionLattice called vtkCGNSSubsetInclusionLattice and uses it to
118 store the SIL. vtkCGNSSubsetInclusionLattice has CGNS-friendly selection API
119 e.g. `vtkCGNSSubsetInclusionLattice::SelectBase(basename)`,
120 `vtkCGNSSubsetInclusionLattice::SelectFamily(familyname)` etc. Readers should
121 provide such subclasses as needed to make it easier on the users. At the same
122 time, the reader can expose API itself that simply forwards to the internal SIL
123 e.g. `vtkCGNSReader::SetBaseArrayStatus` simply calls
124 `vtkCGNSSubsetInclusionLattice::SelectBase` or
125 `vtkCGNSSubsetInclusionLattice::DeselectBase` on the SIL.
126 
127 Once the SIL has been built, it should be placed in the output information
128 using `vtkSubsetInclusionLattice::SUBSET_INCLUSION_LATTICE()` key.
129 
130 A typical `RequestInformation` implementation will have the following form:
131 
132 @code{cpp}
133 int vtkReader::RequestInformation(
134  vtkInformation*, vtkInformationVector**, vtkInformationVector* outputVector) override
135 {
136 
137  if (/*filename changed*/)
138  {
139  /* read file meta-data*/
140 
141  // preserve current selection state on SIL, if any.
142  auto sel = this->SIL->GetSelection();
143  this->SIL->Initialize();
144 
145  /* build SIL */
146 
147  if (sel.empty())
148  {
149  /* setup some default selection to avoid reading nothing by default */
150  this->SIL->SelectAll("//Grid");
151  }
152  else
153  {
154  this->SIL->SetSelection(sel);
155  }
156  }
157 
158  vtkInformation* outInfo = outputVector->GetInformationObject(0);
159  outInfo->Set(vtkSubsetInclusionLattice::SUBSET_INCLUSION_LATTICE(), this->SIL);
160  return 1;
161 }
162 @endcode

164 Using a SIL to define selections
165 --------------------------------
166 
167 Readers that support a SIL for block selection need to expose the SIL so that
168 users can use it to make selections. Typically, you'd expose the SIL by a simple
169 `Get` method.
170 
171 @code{cpp}
172 class vtkReader : public vtkAlgorithm
173 {
174  ...
175 public:
176 
180  vtkSubsetInclusionLattice* GetSIL() const
181  { return this->SIL; }
182 
183 };
184 @endcode
185 
186 For readers that define a subclass, you would want to change the `GetSIL`
187 signature to return an appropriate type rather than the generic
188 vtkSubsetInclusionLattice.
189 
190 Additionally, readers should expose selection API for clearing selections and
191 selecting/deselecting nodes by path. This is useful not only for the users to
192 make easy selections, but also when defining ParaView Server-Manager
193 configuration XML.
194 
195 @code{cpp}
196 class vtkReader : public vtkAlgorithm
197 {
198 public:
199  ...
200 
201  void ClearBlockStatus()
202  { this->SIL->ClearSelections(); }
203 
204  void SetBlockStatus(const char* path, bool enabled)
205  {
206  if (enabled)
207  {
208  this->SIL->Select(path);
209  }
210  else
211  {
212  this->SIL->Deselect(path);
213  }
214  }
215 
221  vtkIdType GetSILTimeStamp()
222  {
223  return static_cast<vtkIdType>(this->SIL->GetMTime());
224  }
225 };
226 @endcode
227 
228 The names of these methods need not be the same as shown in these snippets. You
229 are free to choose whatever naming scheme that makes most sense for your reader.
230 
231 
232 In a typical scenario, the user uses the reader as follows:
233 
234 @code{cpp}
235 
236  vtkNew<vtkReader> reader;
237  reader->SetFileName(...);
238 
239  // call UpdateInformation to read metadata
240  reader->UpdateInformation();
241 
242  // now specify blocks to read.
243  reader->Select(...);
244 
245 @endcode
246 
247 In practice, however, the user may call the `Select`/`Deselect` API before
248 calling `vtkAlgorithm::UpdateInformation`. To support this use-case, the SIL
249 needs to preserve selection state for nodes even before the structure has been
250 defined. This is currently supported by making Select/Deselect calls
251 automatically add nodes at the specified path if they don't exist. That combined
252 with the `RequestInformation` ensuring that selection state is preserved even
253 after the SIL has been reconstructed address this use-case.
254 
255 Using SIL when reading blocks
256 -----------------------------
257 
258 An important step to making SIL useful in a reader is for the reader to use the
259 SIL to decide which blocks are selected. This simply means that in `RequestData`
260 methond for the reader where it reads blocks, it should check with the SIL,
261 using `vtkSubsetInclusionLattice::GetSelectionState` method or a convenient variant
262 provided by the subclass (e.g.
263 `vtkCGNSSubsetInclusionLattice::ReadGridForZone`).

266 ServerManager configuration
267 ---------------------------
268 
269 Now that the reader supports SIL-based block selection, we need to expose it in
270 the ParaView UI. This is done by adding the following set of properties.
271 
272 @code{xml}
273  <SourceProxy name="Reader" class="vtkReader">
274 
275  ...
276 
277  <IdTypeVectorProperty name="SILTimeStamp"
278  number_of_elements="1"
279  default_values="0"
280  information_only="1"
281  command="GetSILTimeStamp">
282  <Documentation>
283  Indicates the timestamp when the SIL structured was modified.
284  Useful to determine which to rebuild the SIL on the client.
285  </Documentation>
286  </IdTypeVectorProperty>
287 
288  <StringVectorProperty
289  name="Blocks"
290  command="SetBlockStatus"
291  clean_command="ClearBlockStatus"
292  repeat_command="1"
293  number_of_elements_per_command="2"
294  element_types="2 0">
295  <SubsetInclusionLatticeDomain name="array_list" default_path="//Grid">
296  <RequiredProperties>
297  <Property function="TimeStamp" name="SILTimeStamp" />
298  </RequiredProperties>
299  </SubsetInclusionLatticeDomain>
300  </StringVectorProperty>
301 
302  ...
303 
304  </SourceProxy>
305 @endcode
306 
307 The key is to use the vtkSMSubsetInclusionLatticeDomain for the block selection
308 property. The **Properties** panel will use pqSubsetInclusionLatticeWidget and
309 pqSubsetInclusionLatticeTreeModel to render the vtkSubsetInclusionLattice in a
310 Qt widget.
